# 四层记忆系统（Short-term / Session / Procedural / Long-term）

## 解决什么问题

LLM 无状态，而招聘对话是**跨天、跨会话的长流程**（今天聊岗位、明天约面试、下周问结果）。把全部历史塞 prompt 既贵又超窗口，还会淹没关键信息。方案是按"信息的生命周期和形态"分四层，各配不同的存储与淘汰策略。

## 四层结构

| 层 | 内容 | 存储 | 生命周期 |
|---|---|---|---|
| **Short-term** 短期 | 原始对话消息窗口 | Supabase `chat_messages`（永久）读时裁剪 | 读取时按条数+时间窗+**字符总量上限**三重裁剪 |
| **Session** 会话事实 | 结构化事实：期望城市/年龄/意向岗位/已推荐岗位/已邀群 | Redis hash（`factsv2:*`） | session TTL |
| **Procedural** 程序性 | 招聘流程当前阶段（currentStage/fromStage/advancedAt/reason） | Redis | SESSION_TTL |
| **Long-term** 长期画像 | 用户身份画像 | Supabase `profile_facts` jsonb | 永久 |

关键细节：

- Short-term（`short-term.service.ts`）**存读分离**：写入永久落库，"窗口"是读取时的裁剪策略。超字符上限丢最早的消息并告警。历史回查用独立的 `historyWindowSeconds` 而非 sessionTtl——两个语义不同的窗口不共用一个配置。
- Session facts 的每个字段值不是裸值，而是带 `{value, confidence, source, evidence}` 的结构——**置信度参与合并决策**（见 [[04-Facts并发安全与字段级写入]]）。
- Long-term 的 `profile_facts` jsonb 里每个字段同样自带置信度/来源/证据，`source='extraction'` 标记"经沉淀写入"，与人工修正可区分。
- Procedural 层**只存不判**：阶段是否合理是业务层（stage-strategy）的事，记忆层只做覆盖写 + 追溯字段。

## 统一读取与格式化

`memory.service.ts` 提供 recall 统一 API，并发读四层、单层失败降级为空（记忆读失败不阻断回合）。`formatters/` 把结构化事实渲染成 prompt 里的 fact-lines，注入位置见 [[05-Prompt-Section动态组装体系]]。

## 沉淀链路（Session → Long-term）

会话事实是易失的（Redis TTL），有价值的部分要在**会话空闲后**沉淀进长期画像，由 `settlement.service.ts` 完成：

- 触发：空闲检测（不是每轮都沉淀——每轮沉淀又贵又会写放大）
- 分页扫描消息（单页 500 条），摘要 LLM 只取**末尾一段**，防旧会话过长导致 prompt 失控
- **踩坑**：早期版本 activeAt 检测键与判断阈值**共用同一个 TTL**，形成死锁——"能读到 activeAt 时必然还不到超时；到超时时 key 已 expire 读不到"——导致沉淀**从未触发过**。教训：超时检测的信号存储寿命必须长于超时阈值本身。

## 设计权衡（学习要点)

- 为什么不用向量库/RAG 记忆？结构化事实（城市、年龄、意向）用 schema 化字段比 embedding 检索**可靠且可审计**；本场景事实量小、字段可枚举，RAG 是杀鸡用牛刀且引入不确定性。
- 为什么四层而不是一层？生命周期不同（一轮 vs 一个会话 vs 永久）、写入方不同（原始消息 vs LLM 提取 vs 沉淀）、失效成本不同。混在一起会让 TTL 策略互相打架——settlement 死锁就是 TTL 语义混用的直接教训。
