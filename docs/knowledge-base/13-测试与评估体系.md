# LLM 应用的测试与评估体系

## 问题：传统测试金字塔对 LLM 失效

单测能覆盖确定性代码（合并原语、守卫规则、分段器），但"Agent 回复质量"不是断言能表达的——同输入不同输出、"好"是模糊的多维标准。

## 本项目的分层方案

| 层 | 手段 | 覆盖什么 |
|---|---|---|
| 单测 | Jest（tests/ 镜像 src/ 结构） | 确定性逻辑：facts 合并、守卫规则、消息切分 |
| 回归测试集 | test-suite 域 + Dashboard 页面 | 历史 badcase 固化为用例，修复后批量重放 |
| LLM 评分 | evaluation 域（evaluate 角色模型） | 对话质量多维打分 |
| 生产抽样 | badcase 分析 skill（人机协作 SOP） | 抽生产对话 → 归因 → 修复 → 生成针对性测试集验证 |
| 影子运行 | 守卫 shadow 模式 | 新规则先观测判例再启用 |

## 关键经验

1. **badcase 是最宝贵的测试资产**：生产判例带完整上下文（prompt、工具返回、守卫裁决同 traceId 可查），比手造用例真实得多。闭环是：运营标注 badcase → 归因（prompt 问题/工具问题/守卫误拦）→ 修复 → 判例入回归集。
2. **单轮重放有局限**：重放单条消息无法复现多轮状态（记忆、阶段），跨轮问题要靠完整会话级别复现。
3. **评审自动化的坑**：曾发现 turn 评审因 comment 过长"假成功"——评估管道自身也需要被观测。
4. **LLM 评分不能当唯一真相**：评分模型自己会错，用于趋势监控（这周比上周差了？）而非单例裁决。

## 学习要点

"LLM 应用怎么测试"几乎是必问题。答案框架：**确定性部分传统测试、概率性部分评估集 + 影子运行 + 生产观测闭环**——重点讲 badcase 从发现到固化为回归用例的完整闭环，这是大多数团队缺失的。
