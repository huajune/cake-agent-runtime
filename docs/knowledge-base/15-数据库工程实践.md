# 数据库工程实践（Supabase/Postgres）

## 迁移纪律

- 测试/生产双库隔离（`db:push:test` / `db:push:prod`），**先测试后生产**
- SQL 幂等：新建用 `IF NOT EXISTS` / `ON CONFLICT`；但**修改既有定义**（索引/函数）必须 `DROP IF EXISTS + CREATE`——IF NOT EXISTS 只查名字不查定义，会静默跳过（吃过 0 行事故，见 [[14-生产事故复盘集]] 案例 3）
- **迁移与代码发版必须同步**：只发代码不推迁移（或反之）是反复发生过的事故源
- 上线新表后用一条真实写入验证落库，不能只看部署成功

## Schema 设计裁定

- **按维度分表而非宽表**：回合流水 / 执行事件 / 守卫档案 / 复聊触达 / 业务漏斗各自独立表，同 trace_id join——写入方、生命周期、查询模式不同的数据不同居
- **JSONB 存档案、真实列存指标**：大 JSONB（完整 prompt 达 43KB）走 TOAST，高频读取字段必须提列（ttft 事故）
- 字段自带元数据模式：`profile_facts` jsonb 里每个事实带 `{value, confidence, source, evidence}`——比"值一张表、元数据一张表"简单得多，查询也总是一起用
- 大表前瞻：`ops_events` 日增 50 万行时再上分区，不提前过度设计

## Redis 与 Postgres 的分工

判定标准不是"快不快"，而是**丢了算不算事故**：
- 丢了算事故且无 DB 备份的 key（facts / stage / group-task）——这类是"Redis 唯一事实源"，是架构风险清单上的显式条目
- 缓存/协调/队列类（去重、锁、debounce pending）可自愈，丢了最多重复或延迟

有价值的易失数据靠**沉淀**转移到 Postgres（session facts → profile_facts）。

## 连接池

Supabase `max_conns=60` 被 Dashboard 轮询 + 重试风暴打满过全线 522（见 [[14-生产事故复盘集]] 案例 1）。RPC 轮询要有节流，应用重试要有熔断。

## 学习要点

前端转后端最容易被考的就是数据库。这套实践的叙事线：**幂等迁移 → 双环境纪律 → 观测验证落库 → 读写路径分离设计**，每一条都有真实事故背书，比背书本的"三范式"有说服力得多。
