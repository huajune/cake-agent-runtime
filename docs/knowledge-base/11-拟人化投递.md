# 拟人化投递（分段 + 打字延迟）

## 解决什么问题

LLM 一次吐 500 字大段落，候选人瞬间收到一堵墙——一眼假机器人，信任感崩塌直接影响回复率。真人招聘顾问是分几条短消息、有打字间隔地发。

## 实现（`delivery.service.ts` + `typing-policy.service.ts`）

1. **分段**：按 `\n\n` 切段；prompt 里约定模型可用 `~` 做软分隔符标注更细的切分点（MessageSplitter 统一处理）
2. **打字延迟**：`delay = 段落字符数 / typingSpeedCharsPerSec * 1000 * 随机扰动`
   - 首段 **0 延迟**（用户在等回复，第一段要快，拟人化不能牺牲首响）
   - 后续段 clamp 到 [minDelay, maxDelay]，且不低于 `paragraphGapMs`
3. 所有参数（是否分段、打字速度、段间隔）在 `hosting_config` DB 配置，Dashboard 热调
4. **投递前置检查**：命中"已暂停托管"（真人接管中）时整条回复丢弃——真人和 AI 同时说话是重大事故

## 学习要点

- 体现"产品思维进架构"：拟人化不是 UI 细节，而是投递层的一等公民，有独立 policy service 和可运营配置。
- 首段零延迟是关键权衡：拟人化(慢)与响应性(快)冲突时，用"首段快、后续拟人"两全。
- 真人接管检查放在**投递时刻**而非生成时刻——生成要几十秒，期间状态可能变化，与 [[09-复聊独立生成链路]] 的"到点核验"是同一条原则：**副作用执行时刻重新校验前提**。
