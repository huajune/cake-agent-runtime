# 多模型三层容错架构（Registry → Reliable → Router）

## 解决什么问题

生产 Agent 依赖多家 LLM 厂商（Anthropic/OpenAI/OpenRouter 等），任何单一厂商都会出现限流、超时、宕机。不能让一次 API 抖动直接变成候选人收不到回复。同时不同任务对模型的"智力/成本/延迟"要求不同，不该所有场景都用最贵的模型。

## 三层设计

```
Layer 1: RegistryService   — 注册（Vercel AI SDK createProviderRegistry 统一多厂商）
Layer 2: ReliableService   — 容错（错误分类 + 指数退避重试 + fallback 降级链）
Layer 3: RouterService     — 路由（按"角色"而非按"模型"编程）
```

### Layer 2 容错的具体算法（`src/providers/reliable.service.ts`）

三类错误分类（`classifyError`）：
- **non_retryable**：401/403/404/400、invalid api key、model not found、余额不足 → 立即放弃，直接走 fallback
- **rate_limited**：429 → 优先解析响应里的 `Retry-After` 头（cap 30s），没有才用指数退避
- **retryable**：5xx / timeout / 网络错误 → 指数退避重试

默认参数（`types.ts` DEFAULT_RELIABLE_CONFIG）：`maxRetries: 3`，退避 `100ms * 2^(attempt-1)`，上限 `10s`。

重试打光后按 **fallback 链**逐个换模型再试（配置在环境变量里，如 `AGENT_CHAT_MODEL_FALLBACKS`）。

### Layer 3 角色路由（`src/providers/router.service.ts`）

业务代码永远不写死模型 ID，而是声明"我要一个 **chat / extract / vision / evaluate / review** 角色的模型"，由 `AGENT_{ROLE}_MODEL` 环境变量决定实际模型。`resolveRoute` 支持 override 精确模型和显式 fallback 覆盖。

角色划分的依据是任务特征：
- `chat`：主对话，要最强模型
- `extract`：事实提取，结构化任务，可用便宜快模型
- `evaluate` / `review`：评估与守卫审查，独立配置避免"用同一个模型自证"
- `vision`：图片消息识别

## 设计权衡（学习要点）

- **为什么不直接 try/catch 换模型？** 错误分类是关键——401 重试三次纯属浪费且拖慢降级；429 带 Retry-After 时盲目指数退避会过早或过晚。分类 + 差异化策略是从"能跑"到"生产级"的分界线。
- **为什么按角色路由？** 换模型是运维动作不是代码动作。线上某厂商故障时改一个环境变量即可全量切换，不用发版。这也是成本治理的抓手：提取类任务从大模型切小模型只动配置。
