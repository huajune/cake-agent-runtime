# 生产事故复盘集（学习最硬的弹药）

学习中最该吃透的就是"你遇到过最难的线上问题"这类追问。以下都是本项目真实事故，每个都有完整的"现象 → 根因 → 修复 → 防复发"链条。

## 1. 生产全线 522：连接池耗尽（2026-06-04）

- **现象**：所有服务 522，重启无效
- **根因**：Dashboard 重 RPC 轮询 + 应用层重试风暴，叠加打满 Supabase `max_conns=60`。重启后重试风暴立即重新占满池子——**重试放大了故障**
- **修复**：先**断载**（暂停托管停止入流）再重启才恢复
- **教训**：重试必须带退避+熔断，否则重试就是自我 DDoS；池子容量要按"峰值并发×重试放大系数"估；恢复顺序是先降载再重启

## 2. 发版丢在途消息（v5.13.0，2026-06-09）

- **现象**：发版后个别用户消息永久无回复，无任何报错
- **根因链**：SIGTERM 无 drain（处理中回合直接被杀）+ 锁孤悬后检查任务静默跳过 + pending TTL 300s 过期 = 消息静默丢失；卡 processing 的记录最长一天后才被 cron 标 timeout
- **修复**：锁冲突不再裸跳过（补建重检任务 + pending 续期），详见 [[02-消息debounce聚合与租约锁]]
- **教训**：优雅停机是分布式消费者的必修课；"静默跳过"的每个分支都要问"如果没有下一次触发怎么办"

## 3. upsert 全部静默失败：IF NOT EXISTS 的坑

- **现象**：`guardrail_review_records` 生产 0 行，测试库正常
- **根因**：迁移用 `CREATE INDEX IF NOT EXISTS` 建 trace_id **unique** 索引，但生产库已存在**同名但非 unique** 的草稿索引——IF NOT EXISTS 只查名字不查定义，静默跳过；upsert `ON CONFLICT` 找不到唯一约束**全部失败**
- **教训**：改既有索引/函数定义必须 `DROP IF EXISTS + CREATE`；**上线新表后要用一条真实写入验证落库**，部署成功 ≠ 功能正常

## 4. 列表页 52 秒：JSONB TOAST detoast

- **现象**：消息处理页月度统计 RPC 实测 52.6s
- **根因**：ttft 指标埋在平均 43KB 的 `agent_invocation` JSONB 里，统计要逐行解压（detoast）整个 JSONB 才能取一个数字
- **修复**：ttft 提升为真实列，写入侧直接落列，读取三处改列
- **教训**：Postgres 大字段存 TOAST，"只取一个 key"也要解压整个值；**JSONB 存档案、真实列存高频指标**

## 5. 沉淀从未触发：TTL 死锁

settlement 的空闲检测键与超时阈值共用同一 TTL——"能读到时必然没超时，超时时必然读不到"，功能上线以来**从未执行过**且无人发现。教训：低频异步功能必须有"多久没跑了"的监控；见 [[03-四层记忆系统]]。

## 6. 破冰率恒 100%：指标造假的另一种形式

friend.added 事件靠"收到消息"反推——能收到消息的必然已加好友，分母分子恒等。教训：**指标的分母事件必须独立采集**，从结果反推源头事件的指标是同义反复。

## 学习要点

讲事故的结构：现象（业务影响量化）→ 排查路径（怎么缩小范围）→ 根因（要讲到机制层）→ 修复（短期止血 + 长期根治分开讲）→ 防复发（监控/流程/架构改进）。有防复发这一段才是 senior 级答案。
