# 复聊（Reengagement）独立生成链路

## 解决什么问题

候选人聊到一半消失是招聘漏斗最大的流失点。需要**主动**二次触达：面试提醒、报名信息没填完的跟进、面试后回访。这与"被动应答"是完全不同的问题——没有用户消息作为触发器，出错时也没有用户纠错的机会。

## 链路结构

```
anchor.service        — 锚点：对话中的可跟进事件（约面成功/报名未完成…）注册锚
follow-up-scheduler   — 调度：按场景规则算触达时点，注册 Bull 延迟任务
follow-up.processor   — 到点执行：先做核验（防陈旧任务）再生成
proactive-composer    — 生成：双档
reengagement-delivery — 投递
touch-ledger          — 全生命周期落 reengagement_touch_records
```

## 核心裁定：不复用主 Generator

主 generator 是"带全套工具的多步 Agent"，用它生成一句"明天面试记得带身份证哦"是**杀鸡用牛刀且不可控**（可能自由发挥调工具、产生新副作用）。复聊用独立的 ProactiveComposer，**双档设计**（`proactive-composer.service.ts`）：

- **模板档**：事实齐全的场景（如 interview_reminder，时间地点都是结构化数据）直接确定性模板渲染，零 LLM 成本、零幻觉风险
- **轻 LLM 档**：需要话术的场景（如 booking_incomplete 委婉催填）用**一次性 completion 调用**（无工具、无多步），prompt 由场景注册表（scenario-registry）的 objective + generationPolicy 拼成

两档产物都过 `validate` 校验（长度/禁词/场景匹配），LLM 档生成失败或校验不过**降级放弃本次触达**——主动消息宁可不发，不能发错。

## 防陈旧：到点核验

延迟任务注册后世界会变：面试可能已被取消/改期。processor 到点执行时**重新核验锚点事实**（查当前真实状态），不符则放弃。早期漏了取消/改期核验，出现过"面试已取消还发提醒"的 badcase。

教训：**延迟任务的决策依据必须在执行时刻重新获取**，注册时刻的快照不可信。

## 全生命周期追溯

每次触达从调度、生成（记录用了哪档）、投递到结果全部落 `reengagement_touch_records`，Dashboard 有候选人视角页面。主动触达没有用户反馈兜底，**观测是唯一的质量抓手**。

## 学习要点

- "被动应答 vs 主动触达"的风险剖面完全不同：被动场景用户会纠错，主动场景一条错误消息就是打扰甚至事故——所以生成策略反而要**更保守**（模板优先、LLM 降级放弃）。
- 双档本质是按"事实完备度"分级选择生成策略，是控制幻觉的架构手段而非 prompt 手段。
