# 出站守卫三档裁决链（Hard Rules → LLM 语义审查 → Sanitizer）

## 解决什么问题

Agent 的回复直接发给真实候选人，幻觉的代价是真金白银：编造薪资、虚假承诺"已帮你报名"、泄露内部信息、就业歧视表述——都可能造成客诉或法律风险。出口必须有一道**独立于生成器**的审查。

## 三档结构（`output-guardrail.service.ts`）

```
回复文本 → ① Hard Rules（确定性规则，毫秒级）
        → ② LLM 语义审查（仅高风险触发，shadow/enforce 双模式）
        → ③ Sanitizer（末端清洗）
        → 最终裁决: pass | revise | replan | block
```

### ① Hard Rules（`rules/` 目录，13 条规则）

确定性代码规则，每条独立文件：booking-claim-errors（虚假报名宣称）、false-promises（假承诺，如 group_promise 拦"已拉你进群"）、job-fact-hallucinations / job-fact-value-mismatch（岗位事实与工具返回值比对）、location-claim-errors、brand-name-errors、insurance-policy-claims、internal-info-leaks、discrimination-leaks、repeated-reply（复读机检测）、visual-message-errors 等。

关键：规则**拿着本回合工具调用的真实返回值**去比对回复文本（如岗位薪资是否与 `duliday_job_list` 返回一致）——这是"用确定性事实校验概率性输出"的模式。

每条规则有 action 级别：enforce（参与拦截+告警）/ observe（只落档案不拦截）——新规则先 observe 观察假阳率再升 enforce，**灰度是守卫规则的生命周期默认路径**。

### ② LLM 语义审查（`llm/semantic-reviewer.service.ts`）

只有**高风险场景**（有副作用宣称/承诺性事实）才触发，控制成本与时延。设计原则：

- **LLM 不能自证**：reviewer 用独立的 review 角色模型（见 [[01-多模型三层容错架构]]），且只能基于"证据包"（GuardrailReviewPacket：工具返回、记忆事实、回复文本）裁决，不允许凭感觉
- **低置信 enforce 结论在代码层强制降级为 observe**——不允许模型"低置信但拦截"
- **失败降级分级**：高风险触发时 reviewer 故障 → fail-close（block，宁可不发）；shadow 模式故障 → fail-open + warning 告警
- **shadow / enforce 双开关**（DB `system_config` 热更新）：shadow 是 fire-and-forget 只观测不拦截，新审查逻辑先跑 shadow 收集判例，误拦率可接受才切 enforce
- 前置校验：Dashboard 开了开关但 `AGENT_REVIEW_MODEL` 没配时会主动告警并关闭 llm 档，防止"配置开了、模型没配、高危场景全部 fail-close" 的事故

### ③ 裁决聚合与档案

规则决策按优先级聚合：`block > replan > revise > observe > pass`。守卫全程**只读、无副作用、不改写文本**——revise 的实际重写由 runner 的独立链路做（见 [[06-Agent回合编排与TurnFinalizer]]）。所有守卫命中、shadow verdict 与修复过程落 `guardrail_review_records`；运行次数、通过量和 finding code 统计落 `agent_execution_events`；机器判例不自动创建 BadCase，只有人工确认需要修复的问题才进入 BadCase。

## 学习要点

- 三档是**成本-确定性阶梯**：确定性规则免费且零假阳（构造好条件的话）先挡大头；LLM 审查贵，只上高风险；末端 sanitizer 兜格式。
- shadow 模式 = 守卫界的灰度发布/影子流量，是把传统发布工程实践移植到 LLM 治理。
- 误拦治理是持续战役，见 [[08-守卫假阳治理与业务裁定]]。
