# 守卫假阳治理与业务裁定

守卫上线后真正的工作量不在"拦住坏的"，而在**别拦住好的**。假阳（false positive）会把正常回复错杀成 block/revise，直接损害转化率。几个有代表性的治理案例：

## 案例 1：语义审查误读 markdown

LLM reviewer 把工具返回的岗位列表 markdown 表格误读，判定回复"编造了岗位"。修复不是改 prompt 求它读仔细，而是**改证据包的呈现格式**——给 reviewer 的证据必须预处理成不易误读的结构。教训：LLM 链路的 bug 优先在**输入侧**修，比在指令侧修可靠。

## 案例 2：group_promise 只拦"完成时态"

产品裁定拉群是**两轮协议**：先征询意向（"我拉你进群好吗？"），用户同意后下一轮才真正调用 invite 工具。守卫最初把征询也拦了（以为是假承诺）。修正后规则只拦**完成时态的假宣称**（"已拉你进群"/"邀请已发送"但实际没调工具），征询意向合法放行。上线首日 7 条假阳全部源于此，当天修复。

教训：守卫规则必须编码**业务协议的时序语义**，不能只做关键词匹配。

## 案例 3：性别/年龄不是歧视信息（业务裁定推翻技术假设）

开发者直觉把"性别/年龄要求"当歧视风险拦截。业务裁定：蓝领招聘里岗位的性别/年龄要求**可以对候选人明说**（这是岗位硬条件，说清楚反而省双方时间），守卫不拦不告警；真正的红线是**户籍/民族/专业**歧视。两条规则随 PR 删除。

教训：安全规则的边界是业务/法务问题，不是工程师拍脑袋的问题。

## 案例 4：precheck 臆造 jobId（输入侧幻觉防御）

空会话上来就说"帮我约面试"时，precheck 环节 LLM 会**凭空编造 jobId 和整张候选人报名表**（幻觉填表）。防御是双层：
1. **空召回闸门**：会话里从未召回过任何岗位时，直接禁止走报名链路（确定性代码，不给 LLM 机会）
2. **booking_jobid_provenance 守卫规则**：报名用的 jobId 必须能追溯到本会话某次工具返回（来源审计）

教训：LLM 会在"缺信息但被要求给结构化输出"时**编造完整可信的假数据**，schema 校验挡不住（格式全对），必须做**来源追溯（provenance）校验**。

## 通用方法论

1. 新规则默认 observe（只记录），观察假阳率再升 enforce
2. 每条机器判例只落 `guardrail_review_records`，人工确认需要修复后再提升到 BadCase
3. 用真实判例回归：修规则后拿历史判例批量重放验证
4. 拦截理由必须结构化（ruleId + evidence），不能只有布尔值——否则无法治理
