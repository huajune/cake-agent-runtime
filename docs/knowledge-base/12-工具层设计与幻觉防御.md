# Agent 工具层设计与幻觉防御

## 工具体系

`src/tools/` + tool-registry：岗位查询（duliday_job_list）、约面/改约/取消、拉群邀请、转人工 handoff、召回历史等。另有 `src/mcp/` 支持 MCP 协议动态扩展工具。工具目录（tool catalog）由守卫层管理——哪些工具在哪些阶段可用是受控的。

## 关键设计经验

### 1. 工具 description 是一等 prompt 资产

与工具语义强绑定的业务约束**写进工具 description**，而不是堆在 system prompt（分层放置原则，见 [[05-Prompt-Section动态组装体系]]）。例如"报名 name 必须真实姓名，微信昵称不算"就属于 booking 工具的 description。

### 2. 确定性兜底优于 prompt 祈祷

多个实测案例证明纯 prompt 约束不可靠：
- 企微备注里有品牌名时应优先推该品牌——纯提示词模型随机忽略，最终在 preparation 阶段用代码解析备注、直接把 brandAliasList 带进岗位查询参数
- 品类词（"咖啡"）应展开为品牌集合走品牌召回，而不是让模型自己发挥成"咖啡师"岗位名

模式：**能用代码在工具入参层面固化的规则，不要指望模型每次都遵守**。

### 3. 外部 API 语义要在工具层抹平

- `storeNameList` 是精确匹配（备注里的门店名经常对不上），`searchJobName` 才是模糊匹配——工具层做了 0 结果时的本地子串兜底
- "windows 为空 = 等通知岗位"：precheck 不能判 date_unavailable，booking 可不带 interviewTime——外部系统的隐式语义必须在工具层显式建模，否则 LLM 会按字面理解出错

### 4. 幻觉防御纵深

- 入口：空召回闸门（会话没召回过岗位就禁报名链路）
- 出口：jobId provenance 校验（报名参数必须可追溯到本会话工具返回），详见 [[08-守卫假阳治理与业务裁定]] 案例 4
- 事实比对：出站守卫拿工具真实返回值比对回复文本（薪资、地点等）

## 学习要点

工具设计的核心命题：**LLM 是不可信的调用方**。传统 API 设计假设调用方读文档、传对参数；给 LLM 用的工具要假设调用方会编造参数、误解语义、忽略约束——所以要入参校验、来源审计、语义抹平三层防御。
