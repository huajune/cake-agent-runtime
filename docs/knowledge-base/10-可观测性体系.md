# 可观测性体系（ALS 上下文 + Tracer + Observer 落库）

## 解决什么问题

LLM Agent 的排障比传统服务难一个量级：同一输入不同输出、失败发生在"模型理解"层面而非异常层面。没有"当时模型看到了什么、决定了什么"的完整档案，badcase 无法归因。

## 架构

```
请求入口 → RequestContextService（AsyncLocalStorage 挂 traceId 等上下文）
        → AgentTracer 埋事件（回合开始/工具调用/守卫裁决/…）
        → CompositeObserver 分发
            ├─ LoggerObserver     → 结构化日志
            └─ PersistingObserver → 落 Supabase agent_execution_events
```

- **AsyncLocalStorage**（`context/request-context.service.ts`）让 traceId 贯穿整个异步调用链而不用层层传参——Node.js 侧等价于 Java 的 ThreadLocal + MDC。这对 Agent 尤其重要：一个回合跨越队列消费、多次 LLM 调用、多个工具的 IO。
- **Observer 模式解耦**：埋点方只调 tracer，落到哪里（日志/DB/未来的 OTLP）由 observer 决定，新增消费端不动业务代码。
- **同 traceId 五表可 join**：`message_processing_records`（回合流水，含完整 prompt）、`agent_execution_events`（执行事件）、`guardrail_review_records`（守卫档案）、`reengagement_touch_records`（复聊触达）、`ops_events`（业务漏斗事件）。排障动线：Dashboard 看流水 → 按 traceId 下钻。

## 关键经验

1. **完整 prompt 必须落库**。`agent_invocation.request.agentRequest` 存全量请求——"模型当时看到什么"是不可再现的（记忆/配置/岗位数据都在变），不存就永远失去归因能力。
2. **代价**：大 JSONB 有性能陷阱——列表页只需要 ttft 一个数字，却因为它埋在 43KB 的 JSONB 里导致逐行 detoast，月度统计 RPC 慢到 52.6s；修法是把高频读取字段提升为真实列。教训：**JSONB 存档案、真实列存指标**，读写路径分开设计。
3. **观测不能只打日志**：shadow 判例、降级、守卫命中必须落库或推飞书告警。日志是辅助——没人主动看日志，"打了日志"不等于"有观测"。
4. **观测 schema 按维度分表**：回合流程、评估、反馈、守卫、复聊各自独立表，不塞一张宽表——写入方、生命周期、查询模式都不同。

## 学习要点

- 这套是"买不到才自建"：当时 LangSmith 等托管方案与自有栈（NestJS + Supabase）集成弱，且要与业务表 join 归因（守卫档案 ↔ 转化漏斗），自建事件表反而是最短路径。
- ALS 是 Node 后端进阶知识点，前端转后端的学习中主动讲清楚"为什么不能用全局变量、为什么不想层层传参"很加分。
