## 2. 范围与风险

### 变更摘要

- 报名链路增加无进行中预约时的事实接地，禁止在本轮没有 `duliday_interview_booking success` 证据时空头宣称“已报名”；真实提交成功后强制播报结果和面试安排。
- “只招暑假工吗”按招聘限制疑问处理，不再误写为候选人的暑假工偏好；候选人改口“长期”时清除既有季节工筛选，让岗位查询恢复长期岗位召回。
- 招聘平台截图中的“发布方 / 发布主体 / 招聘代理”等字段不再覆盖候选人明确表达的雇主品牌。
- 收口 badcase 人工反馈池分层和 T1 复测记录；该历史 T1 批次保持只读，本次另建独立发布回归批次。

### 高风险区域

- **P0：预约事实**——误放行会让候选人基于不存在的报名结果空等；误拦截则会在真实报名成功后漏播报。
- **P0：记忆与岗位筛选**——限制疑问若再次写成季节工偏好，或“长期”未清除旧值，会把真实在招岗位过滤为零并触发重复回复或人工接管。
- **P0：品牌识别**——发布主体覆盖真实雇主会把岗位召回导向错误品牌。
- **P1：模型口径**——岗位、薪资和可用性仍取决于实时外部查询结果；发布后观察回复接地与工具事实一致性。

### 部署顺序与回滚

- 部署顺序：实现 PR 与本底账进入 `develop` → 核对版本元数据完整性 → Release PR 合入 `master` → 自动构建和生产部署 → 健康检查与通知核验。
- 回滚版本：`v10.27.0`。
- 回滚条件：出现无工具成功证据的报名完成口径、真实报名成功后持续不播报、季节工筛选无法清除、截图发布方重新覆盖求职品牌，或生产健康检查失败。
- 回滚动作：回退生产镜像到 `v10.27.0`，保留本批测试执行和日志证据后修复重发。
