## 2. 范围与风险

### 变更摘要

- 出站守卫在 replan 时携带首版草稿，增加结构压扁与结论反转回归闸门，并收敛二审 fail-open 选择策略。
- 修复结算周期与岗位事实审查误判，剥离消息时间标记，补齐守卫修复链路的可观测信息。
- 姓名确认支持“本名吗 / 是我本名 / 姓名+测试手机号”等合法确认路径，减少预约入口误拦截。
- 合同、社保、试用期等岗位字段无法回答时当轮转人工；告警卡片展示岗位数据缺口，`handoff_events` 持久化 `missing_job_info`。
- 修复小时聚合读取 `toolName` 的键名漂移，并幂等回填近 30 天空工具统计。
- 复聊直接复用上一轮候选岗位池展示新岗位时，也建立 `store_presented_no_reply` 锚点；复聊详情抽屉锁定底层滚动。
- 增补语义判定三分法架构方案文档。

### 高风险区域

- **P0：候选人可见回复**——错误的守卫修复选择可能压扁岗位卡、反转有岗/无岗结论，或投递首版中的高风险文本。
- **P0：人工介入副作用**——关键用工事实缺失时必须真实调用 `request_handoff`；短路成功后的空回复是合法终态，且 `missingJobInfo` 必须完整传到底账与告警。
- **P0：预约身份确认**——姓名确认误放行会使用昵称报名，误拦截会让真实候选人重复确认或转人工。
- **P0：数据库先行变更**——两条 migration 已先行应用到测试库和生产库；代码必须兼容新增 nullable 列，小时统计函数必须持续返回非空工具聚合。
- **P1：复聊锚点**——候选池映射过宽会误建复聊任务，映射过窄会漏掉真实新岗位展示。
- **P1：Dashboard / UI**——抽屉滚动锁若未正确清理，会在关闭后继续锁住页面滚动。

### 部署顺序与回滚

- 部署顺序：两条数据库 migration 已先行落库并验证兼容 → #715/#716/#717 合入 `develop` → 合并版本元数据 PR → 将本底账固化为最终版本 → 更新并合并 #709 → 合并正式版本固化 PR → tag / Release / 自动部署 → 健康与关键链路验证。
- 回滚版本：`v10.28.0`。
- 回滚条件：守卫修复出现结构压扁或结论反转、姓名门禁错误放行、关键用工事实未真实转人工、复聊锚点误建/漏建显著增加、生产健康检查失败。
- 回滚动作：回退应用镜像到 `v10.28.0`；保留新增 nullable 列、修正后的聚合函数与已回填数据，避免破坏性数据库回滚；暂停相关复聊任务并保留测试/守卫/告警证据后修复重发。
