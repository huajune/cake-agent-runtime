# Agent 自迭代循环方案（离线批量审核 → 归因 → 提案 → 验证 → 灰度）

> 状态：技术方案 v1（2026-07-08，待评审）
> 定位：与 [guardrail-and-memory-redesign-plan.md](./guardrail-and-memory-redesign-plan.md)（止血版）
> 是两条独立轨道——止血版管"发送链路不再自伤"，本方案管"质量问题自动发现、自动修复提案"。
> 设计原则依据：[rules-vs-semantics-design-philosophy.md](./rules-vs-semantics-design-philosophy.md)
> 的 P1（质量问题的干预点在生成时与离线）、P8（升权靠数据）。

---

## 0. 一句话概括

把语义/正确性审核从"人工全量看"改成"LLM 判官批量初筛 + 人抽检校准"，
把修复从"人工归因改 prompt"改成"agent 产出提案落 testing 版 + 回归验证 + 人一键批准"。
**人保留两个高杠杆动作：每周抽检给判官校准、每次变更批准合入。其余全自动。**

## 0.1 铁律（先于一切实现）

1. **循环的质量上限 = 评估器（判官）的精度。** 判官未过标定考核之前，它的产出只是
   "待审清单"，不允许直接进入归因/提案环节。没有标定的自循环叫无人监督的漂移。
2. **不在发送链路上做任何同步 LLM 审核。** 本循环全部异步、离线、批量，
   零回复延迟、判错零生产后果（在线语义档 enforce 已判废弃，见止血版结论）。
3. **每个修复提案必须自带回归用例**，且必须先落 strategy testing 版跑通回归，
   才能进入人工批准队列。不带验证的提案自动废弃。
4. **prompt/策略变更的 released 合入永远留一个人工批准动作**（分钟级）。
   放权只按第 5 节的路线逐步进行，凭数据不凭感觉。

## 1. 现有资产盘点（本方案零新基建，全是接管子）

| 资产 | 现状 | 在循环中的角色 |
|------|------|---------------|
| `guardrail_review_records` | 生产判例全量归档（首版/重写版/裁决/reason_code） | **采样源 A**：守卫命中与 fail-open 判例，天然的"疑似问题"队列 |
| `message_processing_records` / 会话历史 | 全量对话流水 | **采样源 B**：随机正常轮（判官不能只看守卫怀疑过的，否则永远发现不了守卫的盲区） |
| `SemanticReviewerService` + `GuardrailReviewPacketBuilder` | 在线 enforce 已废弃，代码闲置 | **转岗为批量判官**：证据包构造、五类 finding 的 rubric、置信度输出全部复用，只是从"发送前同步调用"改成"离线批跑" |
| 飞书 badcase 多维表 + 反馈样本池 | 人工提交/复核流程已在运转 | **判官产出的落地通道 + 人工标注的存放处**（标定标签就记在这里，不新建表） |
| `analyze-chat-badcases` 分析流程 | 人驱动的抽样分析 SOP | **归因环节的操作手册**：聚类归因 agent 的提示词直接从这个 SOP 提炼 |
| `evaluation/` 评分框架（llm-evaluation） | 四维加权评分（事实/效率/合规/语气） | **提案验证的量化口径**之一 |
| `biz/test-suite` Agent 测试套件 | 对话回归测试已可跑 | **提案验证的回归闸门** |
| strategy 配置 testing / released 双版本 | Dashboard 可编辑、双状态缓存 | **现成的灰度轨道**：提案自动落 testing，人批准后升 released |

## 2. 循环全景

```
                  ┌──────────── 每周人工抽检 20 条 ──────────┐
                  │            （校准判官，唯一的常态人力①）  │
                  ▼                                          │
采样（守卫判例 + fail-open + 随机正常轮）                      │
  → 批量判官（semantic reviewer 转岗，带证据包）               │
  → 结构化 findings ──────────────────────────────────────────┘
  → 聚类归因（"这 40 次命中是同一个 prompt 缺口"）
  → 修复提案（prompt diff / 规则调整建议 / 新回归用例）
  → 自动验证（落 strategy testing 版 → test-suite 回归 + evaluation 评分）
  → 通过 → 人工批准合入 released（唯一的变更闸门②）
  → 未通过 → 提案废弃，归档原因
```

## 3. 分阶段实施（每阶段独立可用，不做完也有价值）

### 阶段 0：判官标定（先给考卷，再谈权力）

- 从两个采样源抽 150~200 条判例（守卫命中 100 + fail-open 全量 + 随机正常轮 50），
  人工标注真阳/假阳/问题类别，标签记入现有飞书 badcase 多维表（加一列即可）；
- 判官（复用 SemanticReviewerService 的 rubric 与证据包，跑批模式）对同一批出判决；
- 脚本对比输出精确率/召回率报告（`scripts/` 下一个只读脚本，不建服务不建表）；
- **达标线：精确率 ≥85%**。不达标就迭代判官 prompt/rubric，重考；达标前循环停在这一步。

产出：一份判官考卷（长期复用，判官 prompt 每次改动都要重考）+ 第一份精度基线。

### 阶段 1：批量初筛 + 归因（替代人工全量看）

- 每日批跑（v1 用 `scripts/` 手动/cron 脚本即可，不新建常驻服务）：
  昨日采样 → 判官 → findings 去重聚类 → 按问题簇输出清单进飞书 badcase 池
  （复用现有通道，运营和现有 SOP 无感切换）；
- 聚类归因用一次 LLM 调用完成（输入：同簇 findings + 相关 prompt section 原文，
  输出：根因假设 + 建议的干预点——prompt / 规则 / 数据流 / 不修）；
- 人的角色从"全量看对话"变成"看聚类后的问题簇清单"——工作量降一个数量级。

### 阶段 2：提案 + 验证 + 灰度（闭环）

- 提案 agent 对每个确认的问题簇产出：具体 prompt diff（落到 strategy testing 版）
  + 至少一条新回归用例（进 test-suite）；
- 自动跑 test-suite 回归 + evaluation 评分，报告附在提案上；
- 通过验证的提案进入批准队列，人一键 testing → released；
- 每个上线的提案记录问题簇 ID，两周后自动复查该簇命中是否下降（用
  `guardrail_review_records` / badcase 池直接 SQL，不建报表）。

## 4. 人的两个常态动作（自动化的边界）

| 动作 | 频率 | 耗时 | 为什么不能省 |
|------|------|------|--------------|
| ① 抽检 20 条给判官校准 | 每周 | ~1h | 判官精度是整个循环的上限；抽检是唯一的外部真值来源 |
| ② 批准提案合入 released | 每提案 | 分钟级 | prompt 变更直接改生产行为；一次批准换掉"判官偏见自动写进生产"的尾部风险 |

## 5. 放权路线（凭数据逐步撤人）

1. 判官精确率连续 4 周 ≥85% → **低风险类自动生成提案**：回归用例草稿、
   observe 规则的退场建议可免逐条生成确认；BadCase 仍须人工确认后提升；
2. 某问题簇的提案连续 3 次"验证通过 + 上线后命中确实下降" → 该簇的同类后续提案
   进入**快速通道**（批准从逐条审改为周批）；
3. prompt 变更的人工闸门**最后放**，且只在回归集规模与判官精度都有量化证据后再议。

## 6. 非目标

- 不做在线（发送链路）语义审核——已判废弃，理由见止血版与哲学文档；
- 不新建 Dashboard、统计 RPC、评估服务；分析一律直接 SQL / 脚本报告；
- 不新建判例存储表——机器判例继续使用 `guardrail_review_records`；人工确认的问题才提升到 BadCase；
- 不做判官模型微调/蒸馏——量级不够，prompt + rubric 迭代足矣。
