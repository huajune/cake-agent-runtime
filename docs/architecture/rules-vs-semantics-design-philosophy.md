# 确定性规则与语义理解的分工哲学

> 状态：设计原则基线（2026-07-07）
> 起因：对全项目 110+ 条规则机制的审计，以及 `guardrail_review_records` 上线首两日（2026-07-06 ~ 07-07）
> 生产数据复盘。本文档回答一个问题：**什么事情该交给确定性系统（正则/词库/闸门），
> 什么事情该交给 LLM 或交互设计，以及为什么。**
>
> 配套执行文档：[guardrail-and-memory-redesign-plan.md](./guardrail-and-memory-redesign-plan.md)

---

## 0. 一句话版本

**确定性系统只该做两件事——守护不可逆的行为、执行封闭形态的匹配；
一切对开放自然语言的理解，要么交给 LLM（带 schema、带置信度、带兜底），
要么通过数据流或交互设计让这个理解变得不必要。**

---

## 1. 问题的本质

本项目中所有"规则打不过语言"的痛点（出站守卫的豁免正则军备竞赛、记忆提取的
词库补丁、修复回路把好回复改坏）共享同一个根因：

**用低带宽的确定性系统（正则、词库、阈值）去裁决高带宽的概率系统（LLM 生成的自然语言），
且裁决结果具有高代价（丢弃回复 / 覆盖事实）。**

正则能表达的语言空间是固定的超平面；LLM 的输出是随模型版本、prompt、上下文漂移的
分布。每修一个假阳（加一块豁免正则），就开一个新的假阴漏洞——代码里的注释已经如实
记录了这场军备竞赛：`claim-assertion.util.ts` 整个文件存在的理由，就是"『没法帮你
登记报名』被当成可约声称"这类误判；`quota_promise` 的正则已经复杂到需要一段注释解释
为什么"名额还"三个字不能裸匹配。

**这不是实现质量问题，是范式选择问题。** 同样的团队用同样的严谨度写的工具层闸门
（jobId 溯源、precheck 契约）几乎零误伤——因为那些闸门守的是结构化行为，不是语言。

## 2. 生产证据（2026-07-06 ~ 07 守卫档案）

以下数据是本文档所有原则的经验基础，来自 `guardrail_review_records` 表：

- 301 条记录中 180 条（60%）进入 repair 重写回路，**58 条最终 block =
  候选人整轮收不到任何回复**（约 29 次/天）；
- **58 次静默中 28 次（48%）是修复回路自己制造的**：首版命中规则（多为假阳）→
  `toolMode:'none'` 无工具重写 → 模型把想调的工具写成了文本
  （`geocode(address="魁奇路")`）→ `internal_output_leak` 拦下 → 静默。
  守卫先误判、再亲手改坏回复、最后拦掉自己改坏的东西；
- 命中榜前两名（`district_level_distance_claim` 68 次、`group_promise_without_invite`
  52 次）抽样均以假阳为主：前者拦的是"区级定位下报了精确公里数"这种可逆的口径瑕疵，
  后者拦的是产品自己设计的两轮拉群协议前置轮话术；
- 标本案例 id=166：候选人问"松江这边有吗"，首版如实推荐了真实在招岗位（9.4km），
  因公里数命中规则进入重写；重写版把真岗位改口成"**暂时没找到合适的岗位**"——
  为了删一个公里数，修复回路制造了一个事实错误，然后这版又被二审拦掉，候选人
  什么都没收到。一条真实求职线索被系统消灭；
- `proactive_insurance_policy_mention`（P0 block）7 次命中 7 次全拦，抽样全部是
  Agent 如实转述岗位硬性要求（"这岗位需要你有第一份工作的社保"），豁免正则接不住
  "需要**你**有…社保"这种插了主语的变体。

## 3. 判定框架：一条出站拦截规则何时成立

发送前拦截（send-time veto）作为设计手段，必须**同时**满足三个条件：

| 条件 | 含义 | 反例 |
|------|------|------|
| **① 风险不对称** | 误放代价 ≫ 误拦代价。误放是不可逆伤害（法律风险、假承诺成聊天证据、内部实现泄漏）；误拦只是少发一条消息 | "推荐了更远门店没说理由"——误放代价是体验瑕疵，误拦代价是丢一条求职线索，不对称性反向 |
| **② 有 ground truth** | 判定能和结构化事实对账（工具返回字段、真实已投递消息），而不是靠词库猜语义 | "系统状态编造"词库：工具真失败时模型说"系统提交失败"也被判编造——规则根本不知道工具失败了没有 |
| **③ 恢复路径可靠** | 拦截后的修复动作本身不能引入新风险 | "丢弃全文 + 无工具重写"：48% 的静默由此制造 |

**三条缺一即不得进入 veto 档（revise/replan/block），最多 observe。**
这是规则准入的硬测试，写进 catalog 的治理注释里。

## 4. 十条设计原则

### P1 守行为，不守语言（Guard actions, not words）

不可逆性存在于**副作用**（报名提交、拉群、发定位），不在措辞。副作用闸门放在
工具层（本项目的 37 个工具闸门是正面典范）：拦截发生在行为发生前，返回结构化
错误，模型拿到错误还能智能地跟候选人周旋——**零智能损耗**。

语言层面的问题（话术、口径、完整性）本质是**生成质量问题**。在投递前的最后一毫秒
去检查语言是最差的干预点：既改不好（只能拦或重写），又最贵（丢弃整版）。
质量问题的正确干预点：
- **生成时**：prompt、上下文、few-shot、结构化注入；
- **生成后离线**：badcase → 归因 → prompt/数据流迭代。

### P2 按构造约束优于按检查约束（Constrain by construction）

**能用数据流封死的错误，不要用检查器去堵。**

案例：值级对账三规则（时薪/结算/班次错报）存在的原因是模型自由文本转抄工具
markdown 里的数字，抄错了。与其在出口核对数字，不如让数字根本不经过模型的手——
岗位卡由代码从 `rawData` 模板渲染（薪资/班次/距离/要求四个槽位代码填），模型只
负责选岗和写卡片前后的对话文字。"时薪 54 说成 17"在结构上不可能发生，一整类
规则失去存在必要。

判断口径：每当想写一条"值一致性"检查规则时，先问**这个值为什么会被模型转述？
能不能让它由代码直接落到输出里？**

### P3 确定性系统的两个合法角色

1. **不可逆行为的闸门**：jobId 溯源、precheck 契约、城市出处、枚举校验。
   特征：判定对象是结构化数据，不是自然语言。
2. **封闭形态的匹配**：手机号（`1[3-9]\d{9}` 假阳率≈0）、"姓名：X"表单回显、
   URL、JSON 骨架/代码块泄漏、封闭词表的品牌错名（"独立日"）。
   特征：模式空间是**封闭的**——合法形态可枚举，不随语言分布漂移。

在这两个角色里，正则**优于** LLM：更可靠、更便宜、可解释。
出了这两个角色（开放语言的语义、语气、意图、完整性），正则必然进入军备竞赛。

### P4 修复原语按可靠性分级，沉默是最差的输出

拦截后的修复动作，按可靠性从高到低：

1. **确定性文本变换**（transform）：删除/替换违规 span。零风险，适合格式与值类
   问题（如把 `9.4km` 替换为 `大概9公里，按区域位置估算的`）；
2. **定点小修**（targeted edit）：只把违规句喂给模型重写该句，schema 约束输出；
3. **带工具重新规划**（replan）：只留给"事实整个错了"的场景；
4. **无工具全文重写**（rewrite, toolMode:none）：最不可靠的一档——模型在信息更少、
   指令更吓人的条件下重新采样，产物不保证更好，经常更坏（见 §2 数据）。
   若保留，必须显式告知"不可调用工具、不可输出 JSON"，且产物不可用时走 5；
5. **fail-open 兜底**：P1/P2 级违规下，修复不可用就**投递首版 + 告警**，
   或投递预设安全话术。**任何可恢复违规都不允许以静默收场**——
   在约面/收资节点，沉默比任何瑕疵回复对业务的伤害都大。

### P5 教练类规则不进 veto 档

"怎么问城市更好""该列出候选城市""该优先推荐近店""该解释拉群原因"——这些是在
教模型**怎么把话说好**。教练规则的正确形态：
- 进 prompt / stage 策略（生成时引导）；
- 短期进 observe 档时必须有明确消费者和退场条件；无人消费的 observe 是死遥测，应删除；
- 永远不进 veto 档——投递前否决改不好话术，只能消灭回复。

识别特征：规则的 feedbackToGenerator 读起来像话术培训材料（"请改写为列出候选
城市让候选人选择，例如…"）的，就是教练规则。

### P6 规则命中率应趋零，命中即欠债

健康的守卫系统里，每次规则命中都是**生成层的 bug 报告**。正确闭环：
命中 → 归因（prompt 缺失？数据流缺陷？模型能力？）→ 修生成层 → 规则命中率下降 →
规则退化为纯监控。

一条常年稳定命中、稳定误伤的规则不是在工作，是在掩盖生成层的问题没人修。
规则数量单调增长（本项目 catalog 曾到 35 条且每个 badcase 还在加）是治理失效的信号：
**每条新规则都增加假阳面、修复连锁面（规则 A 的 repair 产物触发规则 B）和维护债。**

### P7 开放语言的理解交给 LLM 或交互，不交给正则

- **交给 LLM**：语义审查（semantic reviewer）、事实提取（schema 约束的结构化输出）。
  带三件套：置信度自评 + 低置信降级 observe 的代码层兜底 + 证据引用要求
  （LLM 不能自证，只能基于证据包裁决——这条铁律本项目已写进
  `output-guardrail.service.ts`，是对的，坚持）。
- **交给交互**：报名级字段（姓名/电话/年龄）不该信任任何提取算法，Agent 复述
  确认一次（"电话 138xxxx 对吧？"），低置信提取即升级为最高置信事实。
  **交互侧解决语言歧义的成本是一句话，算法侧是无穷的正则。**
  name gate 的"拒收"语义应演进为"触发确认流"语义。

### P8 观测先行，升档靠数据，降档自动化

- 新规则默认 observe 入场；
- 升 veto 档需要：≥2 周 observe 判例 + 人工标注精确率达标（建议 ≥90% 才可 revise，
  block 另加"封闭形态"要求）+ 通过 §3 三条件测试；
- 已在 veto 档的规则，按 `guardrail_review_records` + 人工抽标持续算精确率，
  低于阈值（建议 70%）**自动降档**，不开会不讨论；
- 升降档不是耻辱，是系统在呼吸。拍脑袋定档才是。

### P9 记忆事实用追加式观测 + 确定性解析器，不用可变字段合并

"一份最新 JSON + deepMerge + 置信度守卫"的存储形态有三个结构缺陷：裁决错误不可逆
且不可审计（历史被覆盖）、同置信度覆盖顺序未定义、时效衰减无机制。

正确形态（event sourcing）：
- 每次提取写**不可变观测记录**：(字段, 值, 来源方法, 原文摘录, 消息ID, 置信度, 时间)；
- "当前事实"不存储，由**解析器**按明确优先级从观测流推导：
  `用户确认 > 表单/结构化自陈 > 规则提取 > LLM高置信 > 系统接口标签 > LLM低置信`；
  同级取最新；跨级冲突不静默覆盖，标记 `needs_reconfirm` 交给对话解决（回到 P7）；
- 收益：可审计（每个值有完整来龙去脉）、可回放（解析器改规则可全量重算）、
  冲突显性化（覆盖 bug 从"静默数据损坏"变成"待确认清单"）。

### P10 只记录必要过程数据，分析直接查 SQL

拦截判例、降级事件、修复结果、fail-open 原因这类**影响线上行为的过程数据**必须可查询；
但不要为了分析先造一套评估系统、Dashboard 或统计 RPC。优先复用已有事实表和档案表，
尤其是 `guardrail_review_records`：把必要字段补进既有链路，用 `reason_code`、
`first_rule_ids`、`revised_rule_ids`、`final_decision` 等稳定字段支撑复盘。

原则是：**记录事实，不预建分析产品。** 要分析时直接写 SQL。只有当某类事实没有任何
合适的既有归属、且会长期参与生产决策时，才考虑新增表。

## 5. 决策流程：新 badcase 来了，放哪一层？

```
一个新 badcase 出现
│
├─ 是副作用做错了（错误报名/错拉群/错发定位）？
│    → 工具层闸门（结构化校验），P1
│
├─ 是模型转述结构化数据时抄错了值？
│    → 优先按构造约束：让该值由代码渲染，不经模型转述（P2）
│    → 封不死的残余，值对账规则（有 ground truth，满足 §3）
│
├─ 是不可逆的内容事故（歧视外露/内部泄漏/无据承诺）？
│    → 出站规则档，但仅当模式是封闭形态；动作 block 前先过 §3 三条件
│
├─ 是话术/口径/完整性/语气问题？
│    → prompt / stage 策略修生成层 + observe 规则攒判例（P5）
│    → 永不 veto；成规模后交语义档评估
│
├─ 是语义级错误（意图误解/阶段错位/承诺语气）？
│    → 语义档（LLM reviewer）：先 shadow，精确率达标再 enforce（P7/P8）
│
└─ 是事实记错了（提取错/覆盖错/过期）？
     → 提取层：封闭形态给正则，开放语言给 LLM 结构化提取（P3/P7）
     → 合并层：观测流 + 解析器（P9）
     → 报名级字段：确认流（P7）
```

## 7. 度量

治理靠这四个指标，全部可从判例库直接计算：

1. **静默率**：`final_decision='block'` 次数/天（区分真 P0 拦截与其他）。
   基线：29/天（2026-07-06~07）。目标：非 P0 静默 < 2/天；
2. **修复制造率**：block 中 `revised_rule_ids` 含 `internal_output_leak`（或
   revise_dangling/empty）的占比。基线：48%。目标：≈0；
3. **分规则精确率**：人工抽标 `guardrail_review_records`，
   命中样本中真阳占比。veto 档规则 < 70% 自动降档；
4. **规则命中趋势**：每条 veto 规则的周命中数。持续横盘不降的规则
   → 生成层欠债清单（P6）。
