Cake Agent Runtime · 回合生命周期

一条消息的
回合之旅

一个候选人在企业微信里随手发来一句话，到 Agent 吐出几段拟人化的回复——中间这趟旅程要穿过 去重、聚合、四类记忆、Prompt 组装、三层模型、多步工具循环、Replay 保护、记忆沉淀。本文把这条流水线一站一站讲清楚。

贯穿全文的案例 · 一句招聘咨询
「你们招收银员吗？工资多少？」
城市 上海 jobIntent 收银员 入口阶段 recommend 候选池 3 个岗位

一颗水珠从企微回调落下，七站之后变成几段带停顿的拟人回复。每一章对准其中一站。

01/ 10 · 回合模型
turn lifecycle

每一次对话，都是「读 → 想 → 做 → 记」的一拍

整个运行时的心跳就一句话：onTurnStart → Compose → Execute → onTurnEnd。先把这个用户的记忆读出来，组装成给模型的简报，跑一轮「模型＋工具」的循环，最后把这一拍学到的东西写回去。像值班医生接诊：先翻病历、再判断、再处置、最后补一笔病程记录。

i
编排只做两件事。AgentRunnerService 自己只负责「调 LLM」和「收尾」；所有准备工作下放给 AgentPreparationService，所有 LLM 请求统一走 LlmExecutorService。
02/ 10 · 入站
AcceptInboundMessageService

门口的签到簿：先验明正身，再放进来

消息从托管平台 POST /wecom/message 进来，第一关不是 Agent，而是一排闸门：自发消息识别 → 过滤 → 去重（Redis 24h） → 写历史 → 记录入站流水。像前台签到——同一个人重复登记的、不该接待的，门口就拦下；通过的才登记入册。全程**立即返回 200 OK**，后面的活儿都异步做。

自发消息

bot 自己发出去的消息回流时，识别出来直接存成 assistant 历史，不当成用户输入。

去重 · Redis 24h

平台可能重投同一条回调，用 24 小时窗口的去重键挡掉重复触发。

写历史 · chat_messages

落到 Supabase，既是短期记忆的来源，也是审计与回放的依据。

!
先回 200，再干活。回调要的是「我收到了」，真正的 Agent 处理被推到异步队列——这样平台不会因为 Agent 慢而超时重投。
03/ 10 · 聚合
SimpleMergeService · debounce

等你把话说完，再下单

真人聊天爱连发：「你们招收银员吗？」「工资多少？」「在上海哦」。如果每条都立刻惊动 Agent，就是三次割裂的回答。debounce 静默窗口像餐厅服务员——你每说一句他都重置计时，直到你停顿足够久（默认 initialMergeWindowMs = 3000ms）才把整桌的菜一次下单。

i
没有「最大聚合数」上限。每条消息注册一个 delay=3s 的 job；只有距最后一条消息真的静默够久的那个 job 才会触发——debounce 天然在用户停顿后收口，不需要人为设条数上限。
04/ 10 · onTurnStart
MemoryService.onTurnStart()

四个抽屉，一起拉开

动笔前，编排层并行拉开四个抽屉：短期（最近 60 条对话）、会话（意向／已推荐岗位／已邀群）、程序（当前在哪个阶段）、长期（带置信度的画像档案）。同时一个「速记员」对当前这句话做规则匹配，划出高置信事实（城市、品牌、年龄、用工形式…）直接塞进 prompt。关键纪律：LLM 自己没有记忆读写权，全部由编排层在回合首尾统一调度。

i
抽屉各有归属服务。短期 ShortTermService、会话 SessionService、程序 ProceduralService、长期 LongTermService；高置信识别里还有专管用工形式的 labor-form.ts（兼职+／小时工／寒假工／暑假工）。
05/ 10 · Compose
ContextService.compose()

给模型的简报，是一摞分层装订的纸

System prompt 不是一坨字符串，而是按 Section 体系一层层叠出来的。最稳定的在上（角色、业务手册、红线），最易变的在中间——runtime-context 把「本轮会变的全部」收在一起：阶段策略、记忆、本轮提示、硬约束、时间、渠道。每装一层就像往简报夹里插一张纸，顺序固定，模型读起来层次分明。

i
策略有「版本」。strategySource 决定读 released（线上默认）还是 testing（测试套件／dashboard 调试）——同一套 Section，喂不同版本的策略配置。
06/ 10 · Providers
Router → Reliable → Registry

三层保险，外加一条备胎链

模型调用从不直接点名某个厂商，而是说「我要一个 Chat 角色的模型」。三层往下翻译：Router 按角色查到 modelId，Reliable 负责重试与降级，Registry 把 provider/model 实例化。主模型连不上？沿 fallback 链换备胎——像航班取消时自动改签下一班，乘客（业务代码）无感。

Layer 3 · Router

按 role 解析路由：AGENT_CHAT_MODEL / EXTRACT / VISION…

Layer 2 · Reliable

重试（指数退避）+ fallback 链降级，按错误类型分流。

Layer 1 · Registry

启动时按环境变量注册 provider/model → LanguageModel 实例。

i
默认参数。maxRetries=3 · baseBackoff=100ms · maxBackoff=10s。错误分三类：401/403/404 这类 non_retryable 不浪费重试直接降级；429 走 rate_limited 尊重 Retry-After；5xx／超时是 retryable。
07/ 10 · Execute
LlmExecutorService · prepareStep

边查边答：一个会自我收敛的循环

模型不是一次答完，而是「想 → 调工具 → 看结果 → 再想」地多步循环（默认 maxSteps = 5）。两个护栏防它原地打转：prepareStep 动态屏蔽——同一个工具调太多次就没收钥匙（典型：duliday_job_list 不断换参扩面）；空文本恢复——查完资料却忘了开口（有 reasoning、有工具结果、正文却空），兜底再补一条候选人可见回复。

!
屏蔽不等于硬停。收紧 activeTools 的同时会在 system 末尾拼一句说明，避免直接 stopWhen 截断导致整轮无回复。skip_reply 也有互斥：本轮已有业务工具调用，就不允许再「沉默」。
08/ 10 · Tools
ToolRegistryService

十一把工具，其中三把「寄出就收不回」

candidate-consultation 场景挂 11 个内置工具。大多数是只读查询，可以反复试；但有三把带不可逆副作用——推进阶段、邀请进群、提交面试预约。它们像已经投进邮筒的信，触发那一刻外部世界就变了，没法假装没发生。这条性质会直接决定下一章 Replay 的走向。

动态扩展。本轮带图片时运行时注入 save_image_description；MCP 工具可通过 registerMcpTool() 运行时叠加到所有场景。带 🔒 的三个登记在 REPLAY_BLOCKING_TOOL_NAMES——下一章见。
09/ 10 · Replay
ReplyWorkflowService · deferTurnEnd

话还没出口，你又补了一句——撕掉重写，但寄出的信例外

Agent 生成期间用户可能又发了新消息。默认想法是：把刚写好但还没发的回复撕掉，合并新消息重跑一次，免得答非所问。但有个例外——如果首次已经动用了不可逆工具（信已寄出），就不能假装没发生，只能把首次回复投递出去。
下面两个开关，亲手拨一下看走向：

生成期间有新消息？
用户在 Agent 还没回完时又发了一条
首次命中不可逆工具？
advance_stage / invite_to_group / interview_booking
投递首次回复

没有新消息 → 直接采纳

pending list 为空，首次回复就是最终答案，显式触发 turn-end，记忆副作用照常固化。

i
为什么要「撕掉记忆副作用」？首次生成会投影出 presentedJobs / facts。若这条被丢弃的回复把它们写进了 session，下一轮 recall 就被污染了。deferTurnEnd=true 让调用方「采纳后才触发」turn-end——而且重跑只跑一次，不二次 replay，杜绝无限循环。
10/ 10 · onTurnEnd
MemoryLifecycleService · DeliveryService

散场之后：整理笔记，再像真人一样把话打出来

回复被采纳，回合开始收尾。onTurnEnd 先 load_previous_state，然后分两支：分支 A 判断是否到沉淀间隔（够久才把会话沉淀进长期档案，否则 skipped）；分支 B 串行写 session——存候选池、从回复投影已展示岗位、LLM 提取事实。每一步都在流水表里留下 success / skipped + durationMs。最后 DeliveryService 把回复分段、加打字延迟，像真人一样一段一段发出去。

i
失败也有归宿。Agent 抛错时统一走降级：发一条降级话术（AGENT_FALLBACK_MESSAGE 或内置 6 条随机）、分级飞书告警、流水标 status = failed。

一句话，记住整趟旅程

入站去重 → debounce 聚合 → onTurnStart 读四类记忆 → compose 分层组装 → 三层模型容错 → 多步工具循环 → Replay 决定采纳还是重跑 → onTurnEnd 写记忆 + 分段投递。

四段各司其职：渠道层守门与节流，编排层读记忆、组 prompt、跑循环，模型层用三层把「角色」翻译成「具体模型」并兜底容错，记忆层在首尾统一读写——LLM 自己从不碰记忆。Replay 是这条链上最反直觉的一环：默认撕掉重写，唯独「信已寄出」时例外。
