# Agent 工作流与工具说明（运营视角）

> **面向读者**：运营同学
> **目的**：让你看懂 AI 招募 Agent 在每一轮对话里"想了什么、查了什么、做了什么"，方便你排查 case、调策略、和客户解释。
> **最后更新**：2026-04-27

---

## 0. 一分钟速览

我们的 AI 招募 Agent 处理的是一件事：**候选人在企微私聊里发来消息 → Agent 想清楚怎么回 → 自动回复 / 自动预约面试 / 自动拉群 / 自动升级人工**。

它不是一个简单的"问答机器人"。每一轮对话，它会：

1. **回忆**：把这个候选人的历史记录、上轮聊到哪、当前在招聘哪个阶段，全部读一遍。
2. **判断**：判断这一轮要做什么（继续问？查岗位？预约？拉群？沉默？升级？）。
3. **行动**：通过"工具"去查岗位库、查地图、提交预约、发企微定位、拉人进群等等。
4. **回复**：组织一段招募者口吻的话发出去。
5. **记下来**：把本轮新得到的信息（候选人偏好、已展示岗位、阶段变迁）写回记忆，下一轮可以用。

## 2. Agent 的"四种记忆"

Agent 不是每轮都从零开始，它有四种记忆：

| 记忆类型 | 内容 | 存储 | 生命周期 |
| -------- | ---- | ---- | -------- |
| **短期记忆** | 最近的对话（默认最多 60 条） | Redis | 跟随会话，2 小时无活跃淘汰 |
| **会话记忆** | 候选人本次咨询的偏好、已展示岗位池、当前关注岗位、活跃预约 | Redis | 1 天无活跃后沉淀到长期 |
| **程序记忆** | 当前所处的招聘阶段 | Redis | 同上 |
| **长期记忆** | 用户档案（姓名、电话、性别、年龄、健康证、学历等）+ 历史对话摘要 | Supabase | 永久 |

**沉淀机制**：候选人 1 天没发消息后，下一次回来时，Agent 会先把上次会话里有价值的信息（偏好、对话摘要）固化到长期档案，再开始新会话。

## 3. 招聘阶段（程序记忆）

Agent 用"阶段"管理一通对话的流程节奏，常见阶段如下（具体由策略配置决定，可以按品牌定制）：

```
recommend         ← 候选人来问岗位 / 推荐合适岗位
   │
   ▼
needs_collection  ← 收集面试所需的信息（姓名、电话、健康证…）
   │
   ▼
interview_booking ← 确认时间、提交预约
   │
   ▼
onboard_followup  ← 面试/入职跟进
```

**阶段切换由 `advance_stage` 工具触发**，候选人提问明显属于另一个阶段时，Agent 必须先调用这个工具再回复。阶段是可跳跃的（比如候选人一上来就要预约，可以直接跳到 booking）。

## 4. 工具清单（按使用频率排序）

每一轮 Agent 最多调用 5 步工具。下面是它能用到的全部工具，**按使用频率从高到低**排列。每个工具都附了：触发条件、输入、副作用（会不会改库 / 发消息 / 报警）。

### 4.1 `duliday_job_list` — 在招岗位查询 ⭐ 最常用

| 项 | 内容 |
| --- | --- |
| **做什么** | 查在招岗位列表，支持按城市、区域、品牌、门店、岗位类型、坐标范围、薪资、班次等多维筛选 |
| **触发条件** | 候选人问任何关于"招什么岗位 / 哪个品牌 / 哪个门店 / 工资多少 / 几点上班 / 有什么要求 / 有什么福利"的问题 |
| **副作用** | 只读，不改任何数据。结果会写入会话记忆（"这一轮给候选人展示了哪些岗位"）|
| **同轮调用上限** | 3 次（防止 Agent 反复换参扩面，浪费时间） |
| **关键约束** | 后端**只做精确关键字匹配**，不会模糊改写。结果数为 0 时 Agent 会先核对字段；只有 1 条时会扩面再查；≥2 条才基于结果回复 |

> 📝 **运营提示**：候选人觉得"我明明在群里看到这个岗位，怎么 Agent 说没有"，多半是品牌/门店/区域名字写法不一致。可以先到 dashboard 用同样关键字查一遍。

---

### 4.2 `advance_stage` — 推进对话阶段

| 项 | 内容 |
| --- | --- |
| **做什么** | 切换当前对话所处阶段（如从 "recommend" 切到 "needs_collection"） |
| **触发条件** | Agent 判断本轮应该进入新阶段（比如候选人确定要预约 → 切到 needs_collection） |
| **副作用** | 写 Redis 程序记忆。会留下"从哪个阶段切到哪个阶段、原因是什么"的审计日志 |

---

### 4.3 `geocode` — 地理编码

| 项 | 内容 |
| --- | --- |
| **做什么** | 把"九亭"、"人民广场"、"陆家嘴"等地名翻译成标准化的省/市/区 + 经纬度 |
| **触发条件** | 候选人提到商圈/地标/街道，且 Agent 准备做"附近岗位推荐" |
| **副作用** | 只读，不改数据 |

---

### 4.4 `recall_history` — 回溯求职历史

| 项 | 内容 |
| --- | --- |
| **做什么** | 查询候选人本次会话之外更早的求职历史摘要 |
| **触发条件** | ① 用户档案非空（说明是回访用户）；② 候选人说"上次"、"之前"、"以前" |
| **副作用** | 只读 |

---

### 4.5 `duliday_interview_precheck` — 面试前置校验

| 项 | 内容 |
| --- | --- |
| **做什么** | 解释岗位的面试规则（可约日期 / 时段 / 硬性筛选 / 要补交什么材料），但**不真正提交预约** |
| **触发条件** | 候选人问"今天可以吗"、"什么时候可以面"、"要准备什么资料" |
| **副作用** | 只读 |
| **重要约束** | 必须先调用这个工具，**才允许**调用 `duliday_interview_booking`。这是为了避免 Agent 直接拿默认值代填 |

---

### 4.6 `duliday_interview_booking` — 预约面试 ⚠️ 不可逆

| 项 | 内容 |
| --- | --- |
| **做什么** | 真正调用预约接口，提交面试时间 + 候选人信息 |
| **触发条件** | ① 已通过 precheck；② 已和候选人确认完所有必填信息（姓名、电话、年龄、性别等） |
| **副作用** | **写库**：创建 recruitment_case 记录（含 booking_id、bookedAt、jobId）；**通知**：私聊监控飞书群推送；**失败时**：自动暂停托管（运营接手） |
| **关键约束** | 严禁用默认值代填（比如自动填"大专"、"有健康证"）。失败最多重试 2 次。**未返回 success 时严禁告诉候选人"已预约成功"** |

> 🚨 **运营提示**：这是不可撤销的操作。一旦调用就会真实创建预约。如果候选人后续反悔需要改时间，必须人工到 dashboard 处理。

---

### 4.7 `invite_to_group` — 邀请加入兼职群 ⚠️ 不可逆

| 项 | 内容 |
| --- | --- |
| **做什么** | 把候选人拉进或邀请进对应城市/行业的企微兼职群 |
| **触发条件** | ① 面试预约成功后立即调用；② 已经查过岗位但确认没有合适匹配，做关系维护 |
| **副作用** | **外部 API**：群成员 < 50 直接拉人，≥ 50 发邀请卡片；**记忆**：写入 invitedGroups 防重复邀请；**告警**：所有目标群都满时飞书通知运营补群 |

> 📝 **运营提示**：群快满（接近 50）时记得提前建新群，否则候选人会收不到邀请。

---

### 4.8 `send_store_location` — 发送门店位置

| 项 | 内容 |
| --- | --- |
| **做什么** | 给候选人发**企微原生位置消息**（不是文字版地址，是可以直接点击导航的那种） |
| **触发条件** | 已经明确是哪个门店，候选人问"地址在哪"、"发个定位"、"怎么走" |
| **副作用** | 调企微 API 发送 LOCATION 消息。失败时会回退成文字地址 |

---

### 4.9 `raise_risk_alert` — 上报风险告警 🚨

| 项 | 内容 |
| --- | --- |
| **做什么** | 候选人辱骂 / 投诉 / 情绪升级时触发人工介入 |
| **触发场景** | ① 辱骂、人身攻击、粗俗表达；② 威胁投诉、举报、维权；③ 情绪连续升级 |
| **副作用** | **暂停托管**（关闭该候选人的 AI 自动回复）；**飞书告警**到运营群；本轮 Agent 自己还会再发一句共情/致歉的话，然后停下来等运营接手 |

---

### 4.10 `request_handoff` — 请求人工接手

| 项 | 内容 |
| --- | --- |
| **做什么** | 面试/入职跟进阶段遇到 Agent 处理不了的场景时申请人工 |
| **触发场景** | ① 候选人找不到门店；② 到店无人接待；③ 预约信息冲突；④ 入职办理异常；⑤ 其他需人工 |
| **前置条件** | 必须存在活跃的预约 case（否则不允许调用） |
| **副作用** | recruitment_case 状态改为 handoff；暂停托管；飞书告警；本轮 Agent 还会发一句衔接话术 |

---

### 4.11 `skip_reply` — 主动沉默

| 项 | 内容 |
| --- | --- |
| **做什么** | 本轮**什么都不发** |
| **触发条件** | 三个条件**同时**满足才允许：① 候选人本轮消息是纯确认词（"好的"、"嗯"、"收到"）且 < 10 字符；② 上一轮 Agent 已经做了明确推进；③ 本轮没有任何新线索 |
| **副作用** | 不发消息但记一笔流水（统计主动沉默率） |

> 📝 **运营提示**：这个工具是为了减少机器味儿。如果候选人只是"嗯嗯"地确认，Agent 一直接话反而显得不自然。

---

### 4.12 `save_image_description` — 保存图片/表情描述

| 项 | 内容 |
| --- | --- |
| **做什么** | 候选人发图（截图、健康证、表情）时，让 Agent 用文字把内容描述下来存进对话历史 |
| **触发条件** | 用户消息含图片或表情 |
| **副作用** | 把"[图片消息] xxx 简洁描述"写回 chat_messages 记录 |

## 5. 典型剧本

下面是几个常见的真实流程。每个剧本都标注了 Agent 实际调用了哪些工具。

### 剧本 A：基础岗位咨询

```
候选人：你们招收银员吗？工资多少？
   │
   ▼
Agent 准备：读记忆，发现是新用户，无历史
   │
   ▼
🔧 调用 duliday_job_list(position="收银员")
   ↓ 返回 3 个岗位
   │
   ▼
Agent 回复：
"我们这边有 3 个收银员岗位～
~ 麦当劳静安寺店：时薪 25，早晚班排班
~ 肯德基徐家汇店：时薪 24，主要是周末
~ ...
你方便的话先告诉我意向区域和时间段哈"
   │
   ▼
✏️ 收尾：写入 lastCandidatePool=[3个岗位]，刷新活跃时间
```

### 剧本 B：从咨询到预约

```
（已经聊了几轮，候选人锁定麦当劳静安寺店）
候选人：那我想约面试，明天可以吗？
   │
   ▼
🔧 advance_stage(nextStage="needs_collection",
                 reason="候选人明确要约面")
🔧 duliday_interview_precheck(jobId=麦当劳静安寺,
                               requestedDate="tomorrow")
   ↓ 返回：明天 10-12 点 / 14-16 点可约，需要健康证
   │
   ▼
Agent 回复：
"好的～明天 10-12 点和 14-16 点都可以约
~ 你方便哪个时段？
~ 另外这个岗位需要健康证哦，你有的话面试当天带上"
   │
   ▼
（候选人发来：10 点吧，我有健康证。我叫张三 138xxxx）
   │
   ▼
🔧 duliday_interview_booking(
     jobId=麦当劳静安寺,
     interviewTime="2026-04-28 10:00:00",
     name="张三", phone="138xxxx",
     hasHealthCertificate=true, ...
   )
   ↓ 返回 booking_id=xxxx，预约成功
   │
   ▼
🔧 invite_to_group(city="上海", industry="餐饮")
   ↓ 拉入兼职群
   │
   ▼
Agent 回复：
"已经帮你约好啦！明天 10 点麦当劳静安寺店面试
~ 顺便拉你进了我们上海兼职群，后面有合适岗位也会发群里
~ 有任何问题随时找我"
```

### 剧本 C：候选人投诉升级

```
候选人：你们这破店啥也不是！我要投诉你们！
   │
   ▼
🔧 raise_risk_alert(riskType="complaint_risk",
                    reason="候选人威胁投诉",
                    summary="...")
   ↓ 暂停托管 + 飞书告警 → 运营立刻能看到
   │
   ▼
Agent 回复：
"非常抱歉给您带来不愉快的体验
~ 我们的同事会在工作时间内第一时间联系您
~ 您先消消气～"
   │
   ▼
（之后候选人的消息不再触发 AI 回复，等运营人工接手）
```

### 剧本 D：候选人到了门店找不到

```
候选人：我到了静安寺，找不到店啊
   │
   ▼
🔧 send_store_location(jobId=麦当劳静安寺)
   ↓ 给候选人发企微原生定位消息
   │
   ▼
（候选人继续：发了定位我也找不到啊）
   │
   ▼
🔧 request_handoff(reasonCode="store_not_found",
                   reason="候选人按定位仍找不到门店")
   ↓ case 改为 handoff + 暂停托管 + 飞书告警
   │
   ▼
Agent 回复：
"我马上让现场的同事联系你～
~ 你先在静安寺地铁口稍等一下"
```

## 6. 关键保护机制（运营要知道）

### 6.1 静默聚合（debounce 3 秒）

候选人快速连发多条消息时，Agent **不会**每条都回。
机制：每条消息进来后等 3 秒；如果这 3 秒内又有新消息，重新计时；直到 3 秒内没有新消息，才把所有累积的消息一起交给 Agent。

> 这就是为什么候选人发完消息有 3 秒"延迟"——这是故意的，避免机器人秒回又被反复打断。

参数 `initialMergeWindowMs` 在 Dashboard 的托管配置里可以调整。

### 6.2 Replay 保护

Agent 在思考的时候（比如 8 秒），如果候选人又发了新消息，原本 Agent 已经准备好的回复怎么办？

- **如果首次工具里有"不可逆"的（预约/拉群/阶段推进）**：直接发出去（已经做了不能撤），新消息留到下一轮处理。
- **如果首次只读了岗位、查了地图**：丢弃首次回复，把新消息合并进来重跑一次。

### 6.3 自动降级

Agent 抛错时（模型挂了、超时、API 错误等），运行时会：
- 发一句安抚的兜底话术
- 飞书告警通知技术
- 记一笔失败流水

候选人不会看到"系统错误"这种字眼。

### 6.4 输入安全

每条候选人消息进入 Agent 前会过一道 prompt injection 扫描。命中时 Agent 会拒绝执行注入指令但继续正常回复，并异步飞书告警。

## 7. 常见运营问题排查

| 现象 | 可能原因 | 排查方向 |
| ---- | -------- | -------- |
| Agent 说"没有合适岗位"，但群里明明在推 | 关键字匹配不上（品牌名/区域名写法不一致） | 用候选人原话到 dashboard 的岗位查询里复现 |
| Agent 一直问同样的问题 | 阶段没推进 / 高置信信号没识别上 | 看流水，确认 `advance_stage` 是否被调用 |
| 候选人连发 5 条 Agent 才回一次 | 是预期行为（debounce 3s 静默聚合） | 看 dashboard 的 initialMergeWindowMs 参数 |
| 候选人骂人 Agent 还在正常回 | 可能没命中 risk 关键字，或 raise_risk_alert 没触发 | 看本轮工具调用日志 |
| 候选人说约好的面试时间不对 | 大概率是 Agent 没确认就直接填了默认值（理论上禁止） | 看 booking 工具的入参，对照前一轮对话 |
| 候选人收不到群邀请 | 目标群已满（≥ 50 直接发邀请卡） | 看飞书告警群是否有"群已满"通知 |
