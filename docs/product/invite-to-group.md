## 业务背景

当前招聘 Agent 以 1v1 私聊为主。兼职群（如"上海零售群"）是岗位信息的二级分发渠道，群内定期推送岗位、抢单信息。将候选人拉入群聊，可以：

1. **弥补匹配缺口** — 当前没有合适岗位的候选人，可以在群里持续关注新岗位
2. **扩大岗位触达** — 已完成登记的候选人，在群里看到更多同城/同行业岗位
3. **降低流失率** — 候选人留在群里，后续有新岗位时可自然触达

群归属于"琪琪"账号（群主），其他托管账号也在群内。拉人使用企业级接口（`/hub-api/api/v1/instantReply/addFromRoom`），不受小组限制，任何 bot 都可跨小组操作。

## 触发策略

Agent 在对话中判断以下两种情况时，调用 `invite_to_group` 工具：

### 策略 1：穷尽推荐后无匹配岗位

> **严格边界**：不是"搜了一次没结果就拉群"，而是 Agent 已尽力为候选人寻找岗位、多次调整推荐策略后仍然无法匹配，才触发拉群。这是最后兜底手段，不是快捷通道。

| 条件 | 说明 |
|------|------|
| 前提 | Agent 已调用 `duliday_job_list` 查询岗位，且经过合理的推荐尝试 |
| 必要条件 | 满足以下**全部**条件才可触发：|

**必须同时满足的条件：**

1. **已明确候选人意向** — 已获取城市、区域等关键信息，不是信息不全导致的搜索无结果
2. **已尝试多维度推荐** — 如：换区域、换品牌、换岗位类型等，而非仅做一次搜索
3. **确认无可推荐岗位** — 在合理范围内（如同城其他区域、相近行业）确实没有匹配项
4. **候选人已知晓现状** — 已告知候选人当前没有合适岗位，而非跳过推荐直接拉群

**禁止触发的场景：**

- 候选人刚说了城市，还没细化到区域/品牌，第一次搜索没结果就拉群
- 候选人提了一个品牌没有，还没尝试推荐其他品牌/岗位
- 对话刚开始，信息收集不完整时

| 话术 | "目前确实没有找到合适的岗位，我帮你加入 XX 群，有新岗位会第一时间推送" |

### 策略 2：登记完成

| 条件 | 说明 |
|------|------|
| 前提 | Agent 已完成面试预约 / 候选人信息收集 |
| 触发 | `advance_stage` 推进到后续阶段 |
| 话术 | "登记成功！我拉你进 XX 群，里面经常有同城的好岗位" |

> **触发由 LLM 判断**：通过工具描述和 system prompt 中的策略指引，LLM 自主决定何时调用。不做硬编码触发。

## 群匹配规则

### 标签体系

群通过托管平台标签标识，格式为 3 级标签：

```
[群类型, 城市, 行业]
```

示例：
| 群名 | 标签 |
|------|------|
| 上海餐饮群 | [兼职群, 上海, 餐饮] |
| 上海零售群 | [兼职群, 上海, 零售] |
| 武汉餐饮群① | [兼职群, 武汉, 餐饮] |
| 武汉餐饮群② | [兼职群, 武汉, 餐饮] |

### 匹配优先级

```
1. 精准匹配：城市 + 行业 完全匹配（如上海+零售 → 上海零售群）
2. 城市匹配：仅城市匹配（如上海+未知行业 → 上海任意兼职群）
3. 无匹配：  告知候选人暂无该城市的群，引导其他方式关注
```

### 同标签多群选择

当同一 城市+行业 存在多个群时（如"武汉餐饮群①""武汉餐饮群②"），按以下规则选群：

```
优先选择人数最少的群（负载均衡）
→ 如果所有群都满，触发群容量管理流程
```

## 群容量管理

企微群存在人数上限（通常 200 人或 500 人，取决于群设置）。系统需要处理群满场景。

### 容量检测

- **数据来源**：`getRoomSimpleList` API 返回的 `memberCount` 字段
- **扩展 GroupContext**：增加 `memberCount` 字段，在群解析阶段写入
- **容量阈值**：配置 `GROUP_MEMBER_LIMIT`（默认 200），预留 buffer

### 群满处理策略

```
匹配到的同标签群列表
  → 按 memberCount 升序排序
  → 选第一个未满的群（memberCount < GROUP_MEMBER_LIMIT）
  → 如果全部已满 → 执行溢出策略
```

### 溢出策略

当某个 城市+行业 的所有群都满时，**对候选人静默处理**，不暴露群满状态：

1. **飞书告警** — 发送告警到运营群，包含：
   - 城市、行业
   - 当前群列表及人数
   - 提示"需要创建新群并打标签"
2. **静默跳过** — 工具返回 `group_full` 状态，Agent **不向候选人提及群满**，正常继续对话
3. **人工介入** — 运营在托管平台手动：
   - 创建新群（如"武汉餐饮群③"）
   - 打上标签 [兼职群, 武汉, 餐饮]
   - 系统下次查询自动识别新群（10 分钟缓存过期后）

> **为什么不自动建群**：托管平台 API 当前不支持建群接口。且建群涉及群名规范、群主分配、群公告设置等运营决策，适合人工处理。

## 核心流程

```
Agent 对话中 LLM 决定拉人进群
  │
  ├─ 输入：city（必填）, industry（可选）
  │
  ▼
invite_to_group 工具执行
  │
  ├─ 1. 获取兼职群列表
  │     GroupResolverService.resolveGroups('兼职群')
  │     （带 10 分钟缓存）
  │
  ├─ 2. 按城市筛选
  │     groups.filter(g => g.city === city)
  │     无匹配 → 返回可用城市列表，LLM 重试或告知用户
  │
  ├─ 3. 按行业精筛（可选）
  │     有 industry → 优先匹配，无匹配回退到城市级
  │
  ├─ 4. 容量排序 & 选群
  │     按 memberCount 升序，选第一个未满的
  │     全部满 → 飞书告警 + 返回 group_full 状态
  │
  ├─ 5. 检查重复
  │     查询 Redis 防止重复拉入同一个群
  │     key: invite:{corpId}:{userId}:{roomWxid}
  │     已拉过 → 返回 already_invited
  │
  ├─ 6. 执行拉人（企业级接口，不受小组限制）
  │     RoomService.addMemberEnterprise({token, imBotId, botUserId, contactWxid, roomWxid})
  │     token = STRIDE_ENTERPRISE_TOKEN（企业级）
  │     imBotId = 聊天 bot 的系统 wxid（从回调透传）
  │
  ├─ 7. 记录 Redis + 写入会话记忆
  │     Redis: invite:{corpId}:{userId}:{roomWxid} = 1, TTL 30天
  │     MemoryService.saveInvitedGroup() → 会话事实记录
  │
  └─ 8. 返回结果
        成功 → { success, groupName, city, industry, inviteMode }
        失败 → { error, reason }
        inviteMode: "direct"（<100人直接拉入）| "link"（>=100人发邀请链接）
```

## 异常处理

| 场景 | 处理方式 | 返回给 LLM |
|------|---------|-----------|
| 无兼职群数据 | 缓存为空 + token 未配置 | `{ success: false, error: "暂无可用群" }` |
| 城市无匹配 | 返回可用城市列表 | `{ success: false, availableCities: [...] }` |
| 全部群满 | 飞书告警，对候选人静默跳过 | `{ success: false, reason: "group_full" }` |
| 已拉过此群 | Redis 去重 | `{ success: false, reason: "already_invited", groupName }` |
| addMember API 失败 | 记录日志 | `{ success: false, error: "拉人失败: ..." }` |
| 候选人已在群中 | API 返回错误/幂等 | `{ success: false, error: "用户已在群中" }` |

## 工具定义

### invite_to_group

```typescript
{
  name: 'invite_to_group',
  description: `邀请候选人加入企微兼职群。根据城市和行业匹配合适的群。

使用场景：
1. 穷尽推荐后无匹配岗位 — 必须同时满足：已明确候选人意向（城市/区域）、已尝试多维度推荐（换区域/品牌/岗位类型）、确认无可推荐岗位、已告知候选人当前没有合适岗位。禁止在信息不全或仅搜索一次无结果时调用。
2. 候选人完成登记后 — 面试预约/信息收集完成，邀请入群获取更多机会。

调用前必须已知候选人所在城市。`,

  inputSchema: {
    city: string      // 必填，候选人所在城市
    industry: string  // 可选，行业偏好（餐饮/零售等）
  },

  output: {
    success: boolean
    groupName?: string     // 拉入的群名称
    city?: string
    industry?: string
    inviteMode?: 'direct' | 'link'  // 拉群方式（<100人直接拉入 / >=100人发邀请链接）
    reason?: string        // 失败原因代码
    error?: string         // 错误信息
    availableCities?: string[]  // 城市无匹配时返回可用列表
  }
}
```

## 数据依赖

### 环境变量

| 变量 | 说明 | 必填 |
|------|------|------|
| `GROUP_TASK_TOKENS` | 小组级 token（群列表查询用） | 是（复用） |
| `STRIDE_ENTERPRISE_TOKEN` | 企业级 token（拉人进群用） | 是 |
| `GROUP_MEMBER_LIMIT` | 群人数上限阈值 | 否，默认 190 |

### 服务依赖

| 服务 | 用途 |
|------|------|
| `GroupResolverService` | 获取 & 筛选兼职群列表 |
| `RoomService` | 执行 addMemberEnterprise 拉人（企业级接口） |
| `RedisService` | 去重记录（防重复拉群） |
| `FeishuAlertService` | 群满告警 |
| `MemoryService` | 拉群后写入会话事实记录 |

### 数据流

```
GROUP_TASK_TOKENS → GroupResolverService → 兼职群列表（含 memberCount）
                                              ↓
候选人 city/industry ──────────────────→ 匹配 & 选群
                                              ↓
STRIDE_ENTERPRISE_TOKEN ───────────────→ RoomService.addMemberEnterprise()
ToolBuildContext.botImId (imBotId) ─────→   （企业级接口，不受小组限制）
ToolBuildContext.userId (contactWxid) ──→
                                              ↓
                                    ┌─────────┴─────────┐
                                    ▼                   ▼
Redis invite:{corpId}:{userId}:{roomWxid}   MemoryService.saveInvitedGroup()
           ← 去重记录 (30天)                  ← 会话事实记录
```

## 运维操作

### 新增群

1. 在托管平台创建群（群主为琪琪）
2. 打标签：[兼职群, 城市, 行业]
3. 等待 10 分钟缓存过期，或调用 API 清除缓存

### 群满处理

1. 收到飞书告警："XX城市XX行业群已满"
2. 创建新群，命名如"XX群②"
3. 打上相同标签
4. 系统自动识别并开始分配

### 监控

- 飞书告警：群满通知
- 日志：`invite_to_group` 工具调用日志
- Redis：可查询 `invite:*` 了解拉群记录
