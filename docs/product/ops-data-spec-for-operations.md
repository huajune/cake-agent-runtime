## 一、我们要解决什么问题

目前运营每天要回答几个问题：

- **"今天每个号的招聘数据怎么样？"** —— 没有汇总，只能逐个对话翻
- **"我们的转化漏斗在哪个环节流失最严重？"** —— 没有完整漏斗视图
- **"为什么这个候选人转人工了？什么原因最多？"** —— 飞书告警有，但没有汇总分析
- **"加好友后，多少候选人真的回复了？"** —— 没追踪过僵尸好友比例

这次产品迭代就是为了把这些数据**结构化、自动化**地提供给运营。

## 三、运营能拿到的三个数据出口

### 出口 1：飞书运营日报（每天晚上 9 点自动更新）

**地址**：飞书多维表格 [运营日报](https://gingjqcjzc.feishu.cn/wiki/JA9wwoysQiUzBJkFJmnchl9XnJg)

**用途**：每天看当天的成绩单，按号查看每个招聘账号的产出。

**表头**（7 列）：


| 列名    | 含义                  | 怎么算                         |
| ----- | ------------------- | --------------------------- |
| 日期    | 报告日期                | 自动                          |
| 小组    | 账号所属小组（琪琪/宇航/南瓜/小祝/盼盼） | 自动                          |
| 添加好友数 | 当天新增了多少个企微好友        | 候选人首条消息时计数（含加好友握手语）        |
| 破冰数   | 当天有多少新好友真的开口聊了     | 候选人首条**真实**消息时计数（排除"我是xx""请求添加你为朋友"等加好友招呼语；"破冰"=不再是僵尸好友） |
| 成功报名数 | 当天预约面试成功的人数         | Agent 调海绵接口预约成功时计数          |
| 邀请进群数 | 当天邀请候选人进招聘群的成功次数    | invite_to_group 工具成功时计数     |
| 通过数   | 当天面试通过的人数           | 海绵工单中"面试成功"且通过时间是今天的工单数     |


**每行 = 某天某个号的全部数据**。同一小组多个号会有多行。

#### 飞书日报使用场景

- **晚上 9 点后**：打开飞书表，看当天的成绩单
- **周报/月报**：复制日报数据做周/月汇总
- **复盘问题号**：哪个号破冰率特别低？哪个号预约数为零？

### 出口 2：Web 转化分析页（实时）

**地址**：Dashboard 菜单下方 → "**转化分析**"

**用途**：看转化漏斗、各账号横向对比、Handoff 原因分布。

#### Block 1：5 个转化率 KPI（一排）

每张卡显示一个百分比 + 同环比变化：


| KPI       | 公式         | 衡量什么               |
| --------- | ---------- | ------------------ |
| **破冰率**   | 破冰数 / 加好友数 | 开场白质量 + 僵尸好友比例     |
| **报名转化率** | 报名数 / 破冰数  | Agent 收资料和约面的能力    |
| **加群率**   | 进群数 / **破冰数** | 报名后引导进群的能力（分母是破冰，见校准 #2）|
| **面试通过率** | 通过数 / 报名数  | 候选人匹配岗位准确度         |
| **整体转化率** | **通过数 / 加好友数** | **端到端漏斗效率（最关键指标，收口到面试通过、不含入职）** |

> ⚠ KPI 卡展示上述 5 个，第三张是**加群率**（不是"入职率"）。后端 `ConversionKpisResponse`
> **没有 `hireRate` 字段**（见校准 #1/#2）：实际是 `breakIceRate / bookingRate / groupInviteRate / passRate / overallRate`，
> 整体转化率（`overallRate`）= 面试通过数 / 加好友数，不再统计入职。


#### Block 2：Cohort 漏斗（同一批人追踪）

**Cohort 维度切换**：

- **加好友 cohort**：选某时间段加好友的候选人，跟踪他们后续走到哪一步
  - 示例："5/21-5/27 加好友的 500 人里，最终 60 个面试通过"
  - **5 个阶段**：加好友 → 破冰 → 邀请进群 → 报名 → 通过（进群在破冰之后，见校准 #4；无"入职"级）
- **报名 cohort**：选某时间段报名的候选人，跟踪他们后续走到哪一步
  - 示例："5/21-5/27 报名的 80 人里，最终 35 个面试通过"
  - **2 个阶段**：报名 → 通过（无"进群/入职"级）

漏斗图直观看出**哪个环节流失最严重**。

#### Block 3：账号维度对比表

每行一个 bot 账号，便于横向对比：


| 账号      | 所属小组 | 新加好友 | 破冰  | 报名  | 进群  | 通过  | 整体转化率 | 状态  |
| ------- | ---- | ---- | --- | --- | --- | --- | ----- | --- |
| gaoyaqi | 琪琪组  | 120  | 85  | 25  | 20  | 18  | 15.0% | 🟢  |
| ZhuDS   | 小祝组  | 110  | 80  | 20  | 15  | 12  | 10.9% | 🟡  |
| ...     | ...  | ...  | ... | ... | ... | ... | ...   | ... |


- 状态指示：🟢 ≥10% / 🟡 5-10% / 🔴 <5%
- 点击表头可排序
- 可按"小组"筛选

> 🚧 **来源渠道维度（规划中，待数据接入）**：后续会支持按"候选人来源渠道"（扫码/活码/投放等）筛选和对比——看哪个渠道不只是量大、而是获客质量高（破冰率/报名率/通过率）。前提是企微"新增客户"回调能透传渠道信息，正在确认；接入前此维度不展示。

#### Block 4：Handoff 原因分布

**Handoff = 转人工**。当 Agent 无法处理某些场景时，会暂停托管 + 飞书告警，让招聘经理介入。

**10 类原因**（校准 #3，会随产品形态调整——增删原因不影响饼图，运营无需配合改动）：


| 原因                          | 含义          |
| --------------------------- | ----------- |
| cannot_find_store           | 候选人找不到门店    |
| no_reception                | 到店无人接待      |
| booking_conflict            | 预约信息冲突      |
| onboarding_paperwork        | 入职办理问题      |
| interview_result_inquiry    | 追问面试结果      |
| modify_appointment          | 改期/取消已预约面试  |
| self_recruited_or_completed | 候选人已被自招/已入职 |
| no_match_or_group_full      | 无匹配岗位 / 群满需维护（新增）|
| system_blocked              | 工具/系统卡死无法自助（新增）|
| other                       | 其他          |


饼图 + 列表展示，可以看出**哪种问题最高发**，从而优化 Agent 或流程。

> 💡 这些分类会随产品稳定增删（比如细分某类、合并低频类）。届时饼图会自动反映新分类，运营侧不用做任何配置；如对某类原因的命名/含义有建议，欢迎反馈。

#### Web 转化分析页使用场景

- **每周复盘**：看本周漏斗各环节转化率变化
- **小组对比**：看哪个小组转化率最高，复制最佳实践
- **服务质量分析**：Handoff 原因分布反映 Agent 哪些场景处理不好
- **产品迭代决策**：哪个环节流失多，就优先优化哪个环节

## 六、运营常见疑问 Q&A

### Q1：为什么"通过数"今天显示 10，而昨天报名只有 5 个？

**A**：因为"通过数"是**当天发生通过事件的数量**，不是"今天报名的人有几个通过了"。今天通过的人可能是 3 天前、5 天前报名的。两个指标不是同一批人。

如果想看"今天报名的最终多少通过"，要看 Web 转化分析页的 **报名 cohort**（5/27 报名的 80 人最终多少通过 = 报名 cohort）。

### Q2：为什么"加好友"的人没回复消息，破冰率上不去？

**A**：可能原因：

- Agent 开场白不够吸引人
- 候选人加好友只是好奇，没真实求职意向
- 候选人忘了

可以从 Web 转化分析页看出：

- 破冰率：本期实际破冰比例
- 同环比：和上周对比有没有恶化

### Q3：为什么同一个小组里两个号差距这么大？

**A**：可能原因：

- 两个号面向的候选人画像不同（地域/品牌偏好）
- 两个号的开场白质量不同
- 两个号被分配的岗位池差异

可以用 Web 转化分析页的**账号维度对比表**直接看到每个号的漏斗各阶段表现，定位问题环节。

### Q4：Handoff 一直很多怎么办？

**A**：去 Web 转化分析页看 **Handoff 原因分布**。如果某种原因占比特别大：

- `cannot_find_store` 多 → Agent 的门店定位能力需要优化
- `modify_appointment` 多 → 看是不是预约时段选得不合理
- `interview_result_inquiry` 多 → 候选人面试后追问太多，可能要主动同步面试结果

### Q5：飞书日报里"邀请进群数"和"成功报名数"的关系？

**A**：两者**不是顺序关系**，是**两个独立动作**。

- 一个候选人可能：报名了但不进群、进群了但不报名、又报名又进群
- 在 Web 转化分析页的 Cohort 漏斗里，"邀请进群"是**破冰之后的运营侧支**（分母=破冰人数，排在"报名"之前展示，见校准 #2/#4），不进线性单调链

### Q6：数据多久会更新？


| 数据            | 更新频率            |
| ------------- | --------------- |
| 飞书运营日报        | 每日 21:00 同步当天数据 |
| 转化分析页 5 大 KPI | 实时（事件发生即更新）     |
| 通过数（入职不再采集）| 每 15 分钟同步一次海绵   |
| Handoff 原因    | 实时              |
| 仪表盘           | 实时（15s/60s 轮询）  |


### Q7：飞书表头能改吗？

**A**：能。但每次改完需要通知开发同步飞书表的字段映射。建议这次先用当前 7 列稳定一段时间。

### Q8：仪表盘上"活跃用户"和"托管用户数"有什么区别？

**A**：两者衡量的是**不同的东西**，不要直接对比：


| 指标        | 含义                             | 统计窗口                    | 数据来源                                       |
| --------- | ------------------------------ | ----------------------- | ------------------------------------------ |
| **活跃用户**  | **最近 1 小时**内有消息往来的去重用户（实时在线脉搏） | 固定近 1 小时，**不随顶部时间范围变化** | 消息处理流水                                     |
| **托管用户数** | 所选时间范围内有托管活动的去重用户（累计盘子）        | 跟随顶部时间范围                | 本日=消息流水；更长范围=托管活跃表 user_activity（保留 365 天） |


- **活跃用户 = "此刻有多少人在聊"**：无论顶部选今天还是近 180 天，它始终只看最近 1 小时，卡片上带「近1h」标记。用来判断当前是否有人在实时对话。
- **托管用户数 = "这段时间一共服务了多少人"**：跟随时间范围累计，范围越长数字越大。
- 所以两个数**通常不相等、也不该相等**——一个是瞬时快照，一个是区间累计。（这和之前"本日两者相同"的旧口径已经不一样，活跃用户已改为近 1 小时实时口径。）
