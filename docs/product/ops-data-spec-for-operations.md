# 蛋糕招聘运营数据体系 · 产品说明

**面向**：运营团队
**目的**：对齐数据口径、看板用法、迭代节奏
**最后更新**：2026-05-28

---

## 一、我们要解决什么问题

目前运营每天要回答几个问题：

- **"今天每个号的招聘数据怎么样？"** —— 没有汇总，只能逐个对话翻
- **"我们的转化漏斗在哪个环节流失最严重？"** —— 没有完整漏斗视图
- **"为什么这个候选人转人工了？什么原因最多？"** —— 飞书告警有，但没有汇总分析
- **"加好友后，多少候选人真的回复了？"** —— 没追踪过僵尸好友比例

这次产品迭代就是为了把这些数据**结构化、自动化**地提供给运营。

## 二、运营能拿到的三个数据出口

### 出口 1：飞书运营日报（每天早上 7 点自动更新）

**地址**：飞书多维表格 [运营日报](https://gingjqcjzc.feishu.cn/wiki/JA9wwoysQiUzBJkFJmnchl9XnJg)

**用途**：每天看昨天的成绩单，按号查看每个招聘账号的产出。

**表头**（7 列）：

| 列名 | 含义 | 怎么算 |
|------|------|--------|
| 日期 | 报告日期 | 自动 |
| 小组 | 账号所属小组（琪琪/宇航/南瓜/小祝）| 自动 |
| 添加好友数 | 当天新增了多少个企微好友 | 候选人加 bot 时计数 |
| 破冰数 | 当天有多少新好友真的回复了消息 | 候选人首次发文本消息时计数（"破冰"=不再是僵尸好友）|
| 成功报名数 | 当天预约面试成功的人数 | Agent 调海绵接口预约成功时计数 |
| 邀请进群数 | 当天邀请候选人进招聘群的成功次数 | invite_to_group 工具成功时计数 |
| 通过数 | 当天面试通过的人数 | 海绵工单中"面试成功"且通过时间是今天的工单数 |

**每行 = 某天某个号的全部数据**。同一小组多个号会有多行。

#### 飞书日报使用场景

- **早上 9:30 上班**：打开飞书表，看昨天的成绩单
- **周报/月报**：复制日报数据做周/月汇总
- **复盘问题号**：哪个号破冰率特别低？哪个号预约数为零？

### 出口 2：Web 转化分析页（实时）

**地址**：Dashboard 菜单下方 → "**转化分析**"

**用途**：看转化漏斗、各账号横向对比、Handoff 原因分布。

#### Block 1：5 个转化率 KPI（一排）

每张卡显示一个百分比 + 同环比变化：

| KPI | 公式 | 衡量什么 |
|-----|------|---------|
| **破冰率** | 破冰数 / 加好友数 | 开场白质量 + 僵尸好友比例 |
| **报名转化率** | 报名数 / 破冰数 | Agent 收资料和约面的能力 |
| **面试通过率** | 通过数 / 报名数 | 候选人匹配岗位准确度 |
| **入职率** | 入职数 / 通过数 | 候选人留存（通过后真的来上班）|
| **整体转化率** | 入职数 / 加好友数 | **端到端漏斗效率（最关键指标）** |

#### Block 2：Cohort 漏斗（同一批人追踪）

**Cohort 维度切换**：

- **加好友 cohort**：选某时间段加好友的候选人，跟踪他们后续走到哪一步
  - 示例："5/21-5/27 加好友的 500 人里，最终 35 个入职"
  - 6 个阶段：加好友 → 破冰 → 报名 → 进群 → 通过 → 入职

- **报名 cohort**：选某时间段报名的候选人，跟踪他们后续走到哪一步
  - 示例："5/21-5/27 报名的 80 人里，最终 35 个入职"
  - 4 个阶段：报名 → 进群 → 通过 → 入职

漏斗图直观看出**哪个环节流失最严重**。

#### Block 3：账号维度对比表

每行一个 bot 账号，便于横向对比：

| 账号 | 所属小组 | 好友 | 破冰 | 报名 | 进群 | 通过 | 整体转化率 | 状态 |
|------|---------|-----|-----|-----|-----|-----|----------|-----|
| gaoyaqi | 琪琪组 | 120 | 85 | 25 | 20 | 18 | 15.0% | 🟢 |
| ZhuDS | 小祝组 | 110 | 80 | 20 | 15 | 12 | 10.9% | 🟡 |
| ... | ... | ... | ... | ... | ... | ... | ... | ... |

- 状态指示：🟢 ≥10% / 🟡 5-10% / 🔴 <5%
- 点击表头可排序
- 可按"小组"筛选

#### Block 4：Handoff 原因分布

**Handoff = 转人工**。当 Agent 无法处理某些场景时，会暂停托管 + 飞书告警，让招聘经理介入。

8 类原因：

| 原因 | 含义 |
|------|------|
| cannot_find_store | 候选人找不到门店 |
| no_reception | 到店无人接待 |
| booking_conflict | 预约信息冲突 |
| onboarding_paperwork | 入职办理问题 |
| interview_result_inquiry | 追问面试结果 |
| modify_appointment | 改期/取消已预约面试 |
| self_recruited_or_completed | 候选人已被自招/已入职 |
| other | 其他 |

饼图 + 列表展示，可以看出**哪种问题最高发**，从而优化 Agent 或流程。

#### Web 转化分析页使用场景

- **每周复盘**：看本周漏斗各环节转化率变化
- **小组对比**：看哪个小组转化率最高，复制最佳实践
- **服务质量分析**：Handoff 原因分布反映 Agent 哪些场景处理不好
- **产品迭代决策**：哪个环节流失多，就优先优化哪个环节

## 三、三个出口的分工

| 出口 | 用途 | 数据频率 | 看什么 |
|------|------|---------|--------|
| 飞书运营日报 | 每日成绩单 | T-1 数据，每日 7:00 自动更新 | 昨天每个号的数据 |

## 四、关键数据来源说明

### 数据来源 1：候选人在企微做的事

| 数据 | 来源 |
|------|------|
| 加好友 | 企微"新增客户"事件回调 |
| 发消息 | 企微消息回调 |
| 加群 | 企微邀请进群结果 |

### 数据来源 2：Agent 自身做的事

| 数据 | 来源 |
|------|------|
| Agent 开场白发送成功 | 我们消息发送服务的回执 |
| Agent 回复消息 | 同上 |
| 推荐岗位 | Agent 调 duliday_job_list 工具的结果 |
| 预约面试 | Agent 调 duliday_interview_booking 工具的结果 |
| 邀请进群 | Agent 调 invite_to_group 工具的结果 |
| 转人工 | Agent 调 request_handoff 工具时记录 |

### 数据来源 3：海绵工单系统

| 数据 | 来源 |
|------|------|
| 面试通过 | 海绵工单 currentStatus = "面试成功" |
| 入职 | 海绵工单 currentStatus = "上岗成功" |
| 候选人姓名/电话/品牌/门店 | 海绵工单详情 |

**关键约定**：海绵那边的工单状态是 source of truth。我们的数据每 15 分钟从海绵同步一次"通过"和"入职"信息。

## 五、运营常见疑问 Q&A

### Q1：为什么"通过数"今天显示 10，而昨天报名只有 5 个？

**A**：因为"通过数"是**当天发生通过事件的数量**，不是"今天报名的人有几个通过了"。今天通过的人可能是 3 天前、5 天前报名的。两个指标不是同一批人。

如果想看"今天报名的最终多少通过"，要看 Web 转化分析页的 **报名 cohort**（5/27 报名的 80 人最终多少通过 = 报名 cohort）。

### Q2：为什么"加好友"的人没回复消息，破冰率上不去？

**A**：可能原因：
- Agent 开场白不够吸引人
- 候选人加好友只是好奇，没真实求职意向
- 候选人忘了

可以从 Web 转化分析页看出：
- 破冰率：本期实际破冰比例
- 同环比：和上周对比有没有恶化

### Q3：为什么同一个小组里两个号差距这么大？

**A**：可能原因：
- 两个号面向的候选人画像不同（地域/品牌偏好）
- 两个号的开场白质量不同
- 两个号被分配的岗位池差异

可以用 Web 转化分析页的**账号维度对比表**直接看到每个号的漏斗各阶段表现，定位问题环节。

### Q4：Handoff 一直很多怎么办？

**A**：去 Web 转化分析页看 **Handoff 原因分布**。如果某种原因占比特别大：
- `cannot_find_store` 多 → Agent 的门店定位能力需要优化
- `modify_appointment` 多 → 看是不是预约时段选得不合理
- `interview_result_inquiry` 多 → 候选人面试后追问太多，可能要主动同步面试结果

### Q5：飞书日报里"邀请进群数"和"成功报名数"的关系？

**A**：两者**不是顺序关系**，是**两个独立动作**。
- 一个候选人可能：报名了但不进群、进群了但不报名、又报名又进群
- 在 Web 转化分析页的 Cohort 漏斗里，"邀请进群"是排在"报名"之后的阶段（同一批报名的人最终多少邀请进群）

### Q6：数据多久会更新？

| 数据 | 更新频率 |
|------|---------|
| 飞书运营日报 | 每日 7:00 同步前一天数据 |
| 转化分析页 5 大 KPI | 实时（事件发生即更新）|
| 通过数 / 入职数 | 每 15 分钟同步一次海绵 |
| Handoff 原因 | 实时 |
| 仪表盘 | 实时（15s/60s 轮询）|

### Q7：飞书表头能改吗？

**A**：能。但每次改完需要通知开发同步飞书表的字段映射。建议这次先用当前 7 列稳定一段时间。

## 六、产品迭代节奏

我们分三阶段交付：

### P0（第 1 周）—— 数据底座 + 关键 bug 修复

完成后：
- 数据采集体系搭建好（候选人加好友、报名、通过、入职都有事件记录）
- 修复"预约成功后系统不知道工单 ID"的历史 bug
- 修复"Agent 误判改期场景"的告警 bug

⚠️ 这一阶段运营**看不到新功能**，但是后面所有功能的基础。

### P1（第 2-3 周）—— 飞书日报上线

完成后：
- 每日 7:00 飞书自动更新昨天数据
- 候选人加好友的瞬间 Agent 会主动发开场白
- request_handoff 不再误报"改期"

**运营关注**：每日 7:00 看飞书运营日报是否正常更新；新好友是否有 Agent 主动打招呼。

### P2（第 4 周）—— Web 转化分析页 + 跨系统埋点

完成后：
- Web 转化分析页可用（漏斗 + 对比 + Handoff 分析）
- 仪表盘加小组筛选
- 关键事件同步上报到 huajune 分析系统

**运营关注**：转化分析页的 5 个 KPI 是否符合预期；Cohort 漏斗各阶段数据是否合理。
