# Agent 质量评测系统 — 设计与使用指南

> 本文档面向团队宣讲，介绍对话测试、用例测试、回归验证三大功能的设计思路和使用方法。

---

## 一、系统全景

### 1.1 为什么需要质量评测？

AI Agent 在招聘场景下需要准确回答岗位信息、处理预约流程、理解用户意图。任何事实性错误（"招人"说成"不招"）或工具调用失误都可能直接影响业务。我们需要一套系统化的测试体系来：

- **开发阶段** → 快速调试，实时看到 Agent 的思考过程和工具调用
- **发版前** → 批量跑预设用例，确保核心场景不出问题
- **持续监控** → 用真实对话回放验证 Agent 表现，量化质量分数

### 1.2 三大功能定位

```
┌─────────────────────────────────────────────────────────────────┐
│                      Agent 质量评测系统                          │
├──────────────┬──────────────────┬───────────────────────────────┤
│   对话测试    │     用例测试      │          回归验证              │
│  (手动调试)   │   (批量跑用例)    │      (真实对话回放)            │
├──────────────┼──────────────────┼───────────────────────────────┤
│  开发/调试    │    发版前验证     │        持续质量监控             │
│  实时交互     │    预设场景覆盖   │        真人对话基准对比         │
│  人工判断     │    人工评审       │        LLM 自动评分            │
└──────────────┴──────────────────┴───────────────────────────────┘
```

| 维度 | 对话测试 | 用例测试 | 回归验证 |
|------|---------|---------|---------|
| **入口** | Dashboard → Agent 对话调试 | Dashboard → 测试/验证集 → 用例测试 tab | Dashboard → 测试/验证集 → 回归验证 tab |
| **数据来源** | 手动输入 | 飞书多维表格（测试用例表） | 飞书多维表格（验证集表） |
| **测试粒度** | 单轮交互 | 单轮，批量执行 | 多轮对话，逐轮执行 |
| **评估方式** | 人工目视 + 反馈标记 | 人工评审（通过/失败 + 归因） | LLM 自动评分（0-100） |
| **结果存储** | 不持久化（会话级） | Supabase + 飞书回写 | Supabase + 飞书回写 |
| **典型用途** | prompt 调优、bug 复现 | 回归测试、发版验收 | 质量基线建立、版本对比 |

## 二、对话测试（Agent 对话调试）

### 2.1 设计思路

对话测试是最轻量的测试方式。开发者可以直接向 Agent 发消息，**实时看到流式响应**，包括 Agent 的思考过程（thinking）、工具调用（tool calls）和最终回复。

核心价值：**所见即所得的调试体验**。

### 2.2 功能特性

#### 流式响应展示
- 消息以流式方式逐字输出，配有打字光标动画
- 基于 Vercel AI SDK 实现，延迟极低

#### 思考过程可视化
展示 Agent 的 `wework_plan_turn` 思考块，包含：
- **阶段**（stage）：当前处理阶段
- **置信度**（confidence）：Agent 对当前判断的信心
- **推理过程**（reasoning）：为什么做出这个决策
- **用户需求**（needs）：识别到的用户诉求
- **风险标记**（risk flags）：潜在风险点
- **策略详情**（strategy）：采取的应对策略

#### 工具调用透明化
每次工具调用以可展开的面板形式呈现：
- 工具名称和调用状态
- 输入参数（JSON 格式）
- 返回结果

#### 上下文注入
支持在左侧面板粘贴历史对话记录，格式为：
```
[03/05 14:30 张三] 你好，我想了解一下岗位
[03/05 14:31 招募经理] 您好！请问您想了解哪个品牌的岗位呢？
```
系统会自动识别第一个发言的用户作为候选人角色。

#### 反馈机制
- 对每条回复可标记 **Good Case** 或 **Bad Case**
- 选择错误归因分类（8 种类型）
- 反馈数据自动写入飞书多维表格，用于后续分析

### 2.3 使用步骤

```
1. 打开 Dashboard → 侧边栏 → "Agent 对话调试"

2. [可选] 在左侧「对话历史」区域粘贴历史记录
   → 为 Agent 提供上下文，模拟多轮对话中途介入的场景

3. 在底部输入框输入消息，点击发送

4. 右侧面板实时展示：
   ├── 思考过程（可展开/收起）
   ├── 工具调用详情（可展开/收起）
   └── 最终回复文本

5. 查看响应指标：耗时统计

6. [可选] 点击 👍 / 👎 按钮提交反馈
   → 弹窗中选择场景分类和错误原因
   → 自动保存到飞书
```

### 2.4 适用场景

- 调试新 prompt / system prompt 改动后的效果
- 复现线上反馈的 badcase
- 验证特定工具调用是否正确触发
- 调试上下文理解问题（注入历史对话后测试）

## 三、用例测试（Scenario Test）

### 3.1 设计思路

用例测试面向 **结构化的回归验证**。测试用例维护在飞书多维表格中，一键导入后批量执行，每个用例经过人工评审判定通过或失败。

核心价值：**标准化的回归测试流程**，确保核心场景在每次改动后不退化。

### 3.2 测试用例结构（飞书表格）

| 字段名 | 说明 | 示例 |
|--------|------|------|
| 用例名称 | 测试场景描述 | 查询深圳蜜雪冰城岗位 |
| 用户消息 | 模拟用户的输入 | 深圳有蜜雪冰城的岗位吗？ |
| 对话历史 | 多轮上下文（可选） | `[03/05 10:00 用户] 你好 ...` |
| 预期输出 | 参考答案（评审参考） | 应调用岗位查询工具，返回深圳蜜雪冰城岗位信息 |
| 分类 | 场景分类标签 | 3-岗位推荐问题 |

### 3.3 完整工作流

```
阶段一：导入                          阶段二：执行
┌──────────────┐                   ┌──────────────────────┐
│  飞书测试用例表  │ ── 一键导入 ──→ │  创建批次 + N个用例    │
│  (多维表格)     │                │  状态: created        │
└──────────────┘                   └──────────┬───────────┘
                                              │
                                    自动提交到 Bull Queue
                                              │
                                   ┌──────────▼───────────┐
                                   │  并发执行（3路并发）    │
                                   │  · 调用 Agent API      │
                                   │  · 记录响应/工具/耗时   │
                                   │  · 超时 2分钟 / 重试 2次│
                                   │  状态: running         │
                                   └──────────┬───────────┘
                                              │
阶段三：评审                                    │
┌──────────────────────────────────────────────▼──┐
│  人工评审                                        │
│  · 逐条查看：用户消息 / 预期输出 / 实际回复 / 工具  │
│  · 判定：✅ 通过  或  ❌ 失败（选择失败原因）        │
│  · 支持批量评审模式（自动跳转下一条）               │
│  状态: reviewing → completed                     │
└──────────────────────────────────────────────────┘
                          │
阶段四：回写               │
┌─────────────────────────▼────┐
│  结果回写飞书                   │
│  · 测试状态：通过 / 失败        │
│  · 最近测试时间                 │
│  · 测试批次号                   │
│  · 失败分类 + 错误原因          │
└──────────────────────────────┘
```

### 3.4 失败原因归类体系

评审失败时，需要选择归因分类，帮助定位问题根因：

| 归因分类 | 说明 | 典型场景 |
|----------|------|---------|
| **工具误触发** | 不该调用却调用了 | 用户说"好的"却触发岗位查询 |
| **工具漏调用** | 该调用却没调用 | 用户问岗位却没查询 |
| **工具参数错误** | 调用了但参数不对 | 品牌名/地区名传错 |
| **回复内容错误** | 回复了但内容不准确 | 给出错误的薪资信息 |
| **未理解用户意图** | 完全理解偏了 | 把昵称当聊天内容 |
| **情绪处理不当** | 对用户情绪处理不好 | 用户不满时仍机械回复 |
| **上下文丢失** | 没记住之前聊过的内容 | 重复询问已回答过的问题 |
| **其他问题** | 无法归类的问题 | — |

### 3.5 使用步骤

```
1. 打开 Dashboard → 侧边栏 → "测试/验证集"
   确保顶部 tab 在「用例测试」

2. 点击「一键测试」按钮
   → 从飞书自动拉取最新测试用例
   → 创建新批次并开始执行

3. 在左侧批次列表观察执行进度
   → 显示进度条和已完成/总数

4. 执行完成后，批次状态变为「评审中」

5. 点击批次 → 右侧显示用例列表和统计数据
   → 通过率、各分类通过率等

6. 点击某个用例 → 打开评审弹窗
   ┌─────────────────────────────────┐
   │  用户消息:  深圳有蜜雪冰城岗位吗？  │
   │  预期输出:  应查询并返回岗位信息     │
   │  实际回复:  [Agent 的完整回复]       │
   │  工具调用:  search_jobs(...)        │
   │  耗时:     3.2s                    │
   │                                    │
   │  [✅ 通过]  [❌ 失败 ▼选择原因]      │
   └─────────────────────────────────┘

7. 批量评审模式：点击「开始批量评审」
   → 自动跳转到下一条待评审用例

8. 全部评审完成 → 批次状态自动变为「已完成」
   → 结果自动回写飞书表格
```

## 四、回归验证（Conversation Test）

### 4.1 设计思路

回归验证将**真实的人工客服对话记录**回放给 Agent，让 Agent 在相同的上下文下重新作答，然后由 **LLM（Claude Sonnet 4.5）自动评分**，量化 Agent 与真人回复的质量差异。

核心价值：**用真实对话建立质量基线**，量化跟踪 Agent 在版本迭代中的表现变化。

### 4.2 与用例测试的关键区别

| 维度 | 用例测试 | 回归验证 |
|------|---------|---------|
| 数据来源 | 人工编写的测试用例 | 真实历史对话记录 |
| 对话轮次 | 单轮 | 多轮（完整对话） |
| 评估方式 | 人工评审 | LLM 自动评分 |
| 评估指标 | 通过/失败 | 0-100 分 + 评级 |
| 上下文 | 可选预设 | 真实对话上下文逐轮累积 |

### 4.3 对话解析与轮次拆分

原始对话记录（来自飞书验证集表）：

```
[03/05 14:30 候选人] 你好，想了解一下深圳的岗位
[03/05 14:31 招募经理] 您好！深圳目前有蜜雪冰城和肯德基在招...
[03/05 14:32 候选人] 蜜雪冰城有什么要求？
[03/05 14:33 招募经理] 蜜雪冰城目前招聘的岗位要求是...
[03/05 14:35 候选人] 可以帮我预约面试吗
[03/05 14:36 招募经理] 好的，我帮您登记预约...
```

系统自动解析为 **3 个测试轮次**：

| 轮次 | 用户消息 | 历史上下文 | 真人参考回复 |
|------|---------|-----------|-------------|
| Turn 1 | 你好，想了解一下深圳的岗位 | （无） | 您好！深圳目前有蜜雪冰城和肯德基在招... |
| Turn 2 | 蜜雪冰城有什么要求？ | Turn 1 的完整对话 | 蜜雪冰城目前招聘的岗位要求是... |
| Turn 3 | 可以帮我预约面试吗 | Turn 1+2 的完整对话 | 好的，我帮您登记预约... |

每轮独立调用 Agent，但**历史上下文逐轮累积**，模拟真实多轮对话场景。

### 4.4 LLM 自动评分机制

使用 **Claude Sonnet 4.5** 作为评估模型，对每轮 Agent 回复打分。

#### 评分规则

```
基础分: 60 分

一票否决（直接 0-20 分）：
  · 事实性错误（"招人"说成"不招"）
  · 关键信息完全错误

严重扣分（每项 -20~30 分）：
  · 遗漏关键信息
  · 逻辑矛盾
  · 未完成用户明确要求的任务

一般扣分（每项 -5~15 分）：
  · 信息不完整
  · 回复效率低（只说"我查一下"）
  · 语气不当

加分项（每项 +5 分）：
  · 主动提供替代方案
  · 信息完整且结构清晰
  · 语气友好专业

优先级: 准确性 > 完整性 > 效率 > 语气
```

#### 评分结果等级

| 等级 | 分数区间 | 含义 |
|------|---------|------|
| 🟢 优秀 EXCELLENT | 80-100 | Agent 回复与真人高度一致 |
| 🔵 良好 GOOD | 60-79 | 主要信息覆盖，表述有差异 |
| 🟡 及格 FAIR | 40-59 | 部分信息一致，需要关注 |
| 🔴 不及格 POOR | 0-39 | 回复内容差异较大，需人工复核 |

**自动评审**：分数 ≥ 60 自动标记为通过，< 60 需要人工复核。

### 4.5 完整工作流

```
阶段一：导入与解析

                                                       │

┌──────────────────────────────────────────────────────▼──┐

│  ├── Turn 1: 用户消息 → Agent 回复 → LLM评分: 85分 ✅     │
│  ├── Turn 2: 用户消息 → Agent 回复 → LLM评分: 72分 ✅     │

│  · 批次通过率                        │
│                                     │

│  · 每条对话的平均分、最低分            │

## 五、数据流与技术架构

### 5.1 整体数据流

```
                    ┌─────────────┐
                    │  飞书多维表格  │
                    │  · 测试用例表  │
                    │  · 验证集表   │
                    │  · 反馈表     │
                    └──────┬──────┘
                     导入 ↓ ↑ 回写
                    ┌──────┴──────┐
     Dashboard ←──→ │   后端服务    │ ←──→ Agent API (花卷)
     (React)        │  (NestJS)   │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              ↓            ↓            ↓
         ┌────────┐  ┌─────────┐  ┌─────────┐
         │Supabase│  │  Redis  │  │Bull Queue│
         │(持久化) │  │ (缓存)  │  │ (任务队列)│
         └────────┘  └─────────┘  └─────────┘
```

### 5.2 数据库表结构

```
test_batches (批次表)
├── id, name, test_type (scenario/conversation)
├── status (created → running → reviewing → completed)
├── total_cases, executed_count, pass_rate
└── feishu_app_token, feishu_table_id

test_executions (执行记录表)
├── batch_id → test_batches.id
├── case_name, test_input (JSONB), expected_output
├── actual_output, tool_calls (JSONB), duration_ms
├── execution_status, review_status, failure_reason
├── [回归验证专用] conversation_snapshot_id, turn_number
└── [回归验证专用] similarity_score, evaluation_reason

test_conversation_snapshots (对话源表，回归验证专用)
├── batch_id → test_batches.id
├── participant_name, full_conversation (JSONB)
├── total_turns, raw_text
└── avg_similarity_score, min_similarity_score, status
```

### 5.3 批次状态机

```
              ┌──────────┐
              │ created  │ ── 开始执行 ──→ ┌──────────┐
              └────┬─────┘                │ running  │
                   │                      └────┬─────┘
                   │                           │ 执行完成
                   │                      ┌────▼──────┐
                   │                      │ reviewing │
                   │                      └────┬──────┘
                   │                           │ 评审完成

                   │                      └───────────┘

## 六、三大功能对比总结

| 维度 | 对话测试 | 用例测试 | 回归验证 |
|------|---------|---------|---------|
| **目的** | 快速调试、定点验证 | 批量回归、发版把关 | 质量量化、趋势监控 |
| **输入** | 手动输入消息 | 飞书测试用例表 | 飞书验证集表（真实对话） |
| **执行方式** | 实时流式 | Bull Queue 异步（3 路并发） | 逐轮顺序执行 |
| **评估方式** | 人眼判断 | 人工评审 | LLM 自动评分 |
| **评估标准** | 无固定标准 | 通过/失败 + 8 种归因 | 0-100 分 + 4 级评级 |
| **结果持久化** | 否 | 是（Supabase + 飞书） | 是（Supabase + 飞书） |
| **适用阶段** | 开发中 | 提测/发版前 | 发版后/持续 |
| **典型场景** | 调试 prompt、复现 bug | 确保核心场景不退化 | 量化 Agent 版本差异 |

## 七、最佳实践

### 7.1 日常开发流程

```
修改 prompt / 配置
      │
      ▼
  对话测试：针对性调试，验证改动效果
      │
      ▼
  用例测试：跑一轮完整测试用例，确保没有退化
      │
      ▼
  回归验证：跑真实对话，确认整体分数未下降
      │
      ▼
  提交发版
```

### 7.2 用例维护建议

- **测试用例表**：按场景分类维护，覆盖 6 大场景（品牌识别、地区识别、岗位推荐、情绪处理、预约流程、其他）
- **验证集表**：定期从线上挑选有代表性的真实对话，特别是包含复杂多轮交互的对话
- **Bad Case 沉淀**：对话测试中标记的 bad case 可以转化为测试用例，补充回归测试覆盖

### 7.3 评审效率

- 使用**批量评审模式**可以大幅提升效率（自动跳转下一条）
- 回归验证中 **≥60 分自动通过**，只需关注低分项
- 关注**最低分**而非平均分，最低分往往暴露最严重的问题
