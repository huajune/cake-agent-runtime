# Ship Release

遵循仓库自身约定，把本地未提交改动安全推进到生产发布完成。

## 工作流

1. 先读取仓库约定。
   - 查找并读取 `AGENTS.md`、贡献指南、包脚本、发版脚本、PR 模板和 CI/发布 workflow。
   - 检查当前分支、remote、默认分支、工作区状态、完整 diff、近期提交风格、tag 和已有 PR。
   - GitHub 仓库需确认 `gh` 已安装且已认证。

2. 确认改动范围。
   - 审阅所有 staged、unstaged 和 untracked 文件，按目的分组。
   - 排除生成文件、调试残留、密钥文件和无关改动。
   - 主动识别数据库 migration/schema/RPC、环境变量、运行时配置、密钥、基础设施和部署脚本变化；即使 diff 没有直接修改 `.env.example` 或 migration，也要从新增配置读取、SQL 调用和失败日志反查是否漏项。
   - 若工作区确有多组改动且无法可靠判断发版范围，暂停并向用户确认；不得擅自丢弃或混入用户工作。

3. 整理和审阅代码。
   - 检查正确性、回归、安全性、异常处理、架构、命名、格式和测试覆盖。
   - 只修复为发布当前范围所明确需要的问题，保持原始意图。
   - 从 diff 提炼用户可感知行为、根因、影响和风险，供 commit 与 PR 使用。

4. 创建并维护发版底账。
   - 每次发布都从 `docs/releases/_template.md` 创建一份底账；准备期命名为 `docs/releases/YYYY/pending-YYYY-MM-DD-pr-NNN.md`，版本确定后重命名为 `vX.Y.Z.md`。
   - 底账必须从本批实际 diff、关联 PR、配置、migration 和外部依赖推导，记录范围、高风险区域、部署顺序、回滚版本、回滚条件和负责人。
   - 针对影响链路策展 P0/P1 回归 case，逐条写清输入/前置状态、期望结果、验证方式、状态和证据；不得复制固定清单凑数，不得把原始 BadCase 直接当正式测试资产。
   - P0 是发布阻断项，必须全部通过；模型行为、UI、外部服务或权限变化不能只用单测替代实际回放/联调。
   - 正式测试资产仍通过 `analyze-chat-badcases` 工作流进入飞书测试集/验证集，并保留来源血缘。

5. 执行校验。
   - 先跑相关定向测试，再跑仓库要求的 lint、format check、typecheck、build 和 CI 等价命令。
   - 若发版范围涉及 Agent 回复、Guardrail、预约/工单、复聊、记忆或 test-suite，用 `analyze-chat-badcases` 作为强制配套工作流，完整执行“正式测试资产质量闸门（如适用）→ 真实 Agent 链路 → 逐条业务评审 → testing/飞书状态收口 → 生产 Dashboard 同步 → 生产 API 全量对账”。不得只跑单测或模型直调后宣称回归通过。
   - Agent 回归必须分开记录 runtime 与业务口径：`execution_status=success` 只代表链路完成；只有 `review_status=passed` 且回复、工具、守卫、终态或副作用证据符合 checkpoint，才可标为业务通过。证据不足或动态 fixture 不可控必须标为 `skipped / 测试资产不可评估`，不能记作通过。
   - 评审动态事实前必须核对 Agent 实际收到的完整运行上下文，包括测试输入、当前预约信息、记忆、工具结果、execution trace/steps 和守卫终态；“本轮没有工具调用”不等于“回复无依据”，也不得只看最终文本就判幻觉。短路工具成功时，空回复应按工具契约评审，不能自动判为内容缺失。
   - 生产 Dashboard 同步是已完成 Agent 回归的默认收口。评审结论稳定后运行项目既有同步命令，并核对 `warnings=[]`、批次数、执行数、评审分布、实际回复、工具调用和评审备注；除非用户明确要求仅本地，未完成同步与生产 API 对账不得勾选 Agent 回归闸口。若因隐私、权限、配置缺失或评审未完成而不能同步，必须明确报告并保持发布阻断状态。
   - 运行 `git diff --check`；格式化或修复后重新审阅最终 diff。
   - 数据库变化需检查迁移命名和顺序、SQL 可重复执行性、锁表/回填风险、RLS/权限、索引、函数签名、向前与向后兼容、回滚策略，以及测试库/生产库 migration status；优先先做只读状态检查。
   - 环境变量变化需检查代码读取点、默认值与 fail-fast 行为、`.env.example`、README/运维文档、CI、Docker/托管平台配置和 GitHub Actions secrets/variables 是否同步；只报告变量名和配置位置，绝不打印值。
   - 明确部署顺序：判断 migration、配置和应用代码谁必须先落地，确保滚动发布期间新旧版本均可运行；若不能兼容，拆成 expand/migrate/contract 多阶段发布。
   - 不得削弱、跳过或伪造检查。范围内失败应定位并修复；外部阻塞需给出证据。

6. 创建实现 PR。
   - 安全同步远端引用，不覆盖本地改动。
   - 若当前位于默认或集成分支，按环境要求的分支前缀创建语义清晰的分支。
   - 显式暂存确认范围内的路径，按仓库风格提交，并 push 设置 upstream。
   - 用户要求完成发版时创建 ready-for-review PR，否则默认 draft。目标指向仓库正常集成分支；正文包含摘要、影响、修复根因和实际校验记录。

7. 推进 PR 合并。
   - 持续跟进 required checks 和 review。失败时读取日志，修复范围内问题，push 后重新观察。
   - 仅在规则允许且所有门禁通过后开启 auto-merge 或合并。
   - 不绕过分支保护、不伪造批准、不强推共享分支、不使用管理员越权，除非用户明确要求该项例外。

8. 跟随仓库正式发版链路。
   - 优先使用仓库现有脚本和 workflow，不另造发版命令。
   - 按项目定义继续处理版本/CHANGELOG PR、集成分支到生产分支 PR、固化 PR、tag、GitHub Release、迁移、制品和部署任务。
   - 版本号确定后，将 pending 发版底账重命名为对应 `vX.Y.Z.md`，并核对底账、CHANGELOG、Release PR 和实际发布范围一致。
   - 用户要求“完成发版”时，应继续跟进自动创建的后续 PR 与 CI，直到发布验证完成。
   - 若含数据库变化，持续跟踪测试库迁移、生产库迁移、应用部署与回填任务的每一步；执行前记录 migration status，执行后再次核对已应用版本和关键查询/函数可用性。生产迁移或回填必须使用项目既有命令；不可逆 SQL、大表重写、长锁或数据删除需单独明确确认。
   - 若含环境变量变化，发布前给出新增/修改/废弃变量清单、目标环境和配置入口，确认所需 secrets/variables 已存在后再部署；不得用占位值覆盖真实配置。需要人工写入或轮换密钥时暂停并请求用户完成或授权。
   - 生产数据库迁移、不可逆数据操作、手动密钥变更或非标准部署覆盖属于高风险动作，需单独明确确认。

9. 验证并汇报。
   - 确认合并提交、版本/tag 或 release 制品、部署 workflow 结果及可用的健康/冒烟检查。
   - 若涉及数据库，核对生产 migration status、关键 schema/RPC 和必要回填结果；若涉及环境变量，核对应用启动日志与配置健康检查是否确认变量已生效，但不得输出变量值。
   - 汇报分支、commit、所有相关 PR、校验、版本/tag、数据库迁移、环境变量清单、部署状态和遗留人工动作。
   - 在版本底账补齐实际 tag、Release URL、部署 workflow、发布时间、生产验证、监控、回滚情况和遗留事项；发布未成功时记录失败阶段和处置，不得把状态标为已发布。

## 操作原则

- 以仓库与 GitHub 的实际状态为依据，不猜测分支或发版拓扑。
- 保留用户全部改动与凭据，不输出密钥，不提交环境文件。
- 用户要求完成发版时，应覆盖正常 CI 等待及仓库自动产生的后续流程。
- 仅在范围存在实质歧义、缺少权限/人工 review、高风险生产变更或外部系统无法安全修复时暂停。

## 发版 Checklist

每次执行都创建并维护以下清单，在 PR 正文或最终汇报中保留实际勾选状态和证据链接；不适用项标记 `N/A` 并说明原因，不得直接省略。

### 提交前

- [ ] 已读取仓库开发、PR、CI 与发版约定
- [ ] 已逐文件确认 staged、unstaged、untracked 范围，无密钥、调试残留或无关改动
- [ ] 已审阅业务逻辑、错误处理、安全、架构与测试覆盖
- [ ] 已识别数据库 migration/schema/RPC/权限/回填变化，或确认 `N/A`
- [ ] 已识别新增、修改、废弃环境变量及运行时配置，或确认 `N/A`
- [ ] `.env.example`、文档、CI、容器与托管配置已同步，或确认 `N/A`
- [ ] 已创建本批 pending 发版底账，并从实际 diff 策展 P0/P1 回归 case
- [ ] 底账已记录配置、migration、外部依赖、部署顺序与回滚方案，或明确 `N/A`
- [ ] 定向测试、lint、format check、typecheck、build、全量测试和 `git diff --check` 已通过
- [ ] 最终 diff 已复核，branch、commit message 和 PR 目标分支符合约定

### 合并前

- [ ] PR 摘要、根因、用户影响、风险、验证记录完整
- [ ] 发版底账 P0 全部通过，P1 无新增阻断性回归，证据链接完整
- [ ] Agent 行为相关改动已完成真实链路、逐条业务评审、testing/飞书状态收口、生产 Dashboard 同步与生产 API 全量对账，或确认 `N/A`；不得用 runtime success 代替业务 passed
- [ ] required checks 全绿，阻塞 review 已处理，最新 commit 已获得所需批准
- [ ] 数据库迁移顺序、兼容窗口、锁表/回填/回滚风险已评估，或确认 `N/A`
- [ ] 环境变量名称、目标环境、配置入口和责任人已列明；所需 secrets/variables 已存在，或确认 `N/A`
- [ ] 已明确 migration、配置、应用部署及回填的先后顺序
- [ ] auto-merge/merge 方式符合分支保护和仓库历史约定

### 生产发布前

- [ ] 版本、CHANGELOG、release PR、tag/制品与目标环境一致
- [ ] pending 发版底账已按最终版本重命名，内容与 CHANGELOG、release PR 一致
- [ ] 已记录测试库与生产库 migration status；待执行迁移已精确确认，或确认 `N/A`
- [ ] 不可逆 SQL、大表重写、长锁、数据删除和生产回填已获明确确认，或确认 `N/A`
- [ ] 生产 secrets/variables 已配置且未在日志或 PR 中泄露，或确认 `N/A`
- [ ] 回滚版本、回滚条件和负责人明确
- [ ] 发布窗口内依赖的外部服务、审批和权限可用

### 发布后

- [ ] 所有实现、元数据、release、固化与回同步 PR 状态符合仓库流程
- [ ] GitHub Release/tag/制品创建成功，部署 workflow 全绿
- [ ] 生产 migration status、关键 schema/RPC/权限和回填结果已验证，或确认 `N/A`
- [ ] 应用启动与配置健康检查确认环境变量生效，且未输出变量值，或确认 `N/A`
- [ ] 健康检查、关键业务冒烟和监控/告警无异常
- [ ] 版本底账已补齐实际发布、生产验证、监控、回滚与遗留事项
- [ ] 已记录实际发布版本、commit、时间、结果和遗留动作
